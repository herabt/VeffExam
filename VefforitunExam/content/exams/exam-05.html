SECTION A: useCallback (Slides 430-455)

Question 1 — Multiple Choice

A developer wraps every function inside their component with useCallback, even functions that are only used inside event handlers and never passed as props. What is wrong with this approach?

  • A) useCallback can only be used with functions passed to child components
  • B) It adds unnecessary overhead — React must compare dependency arrays on every render, and there is no performance benefit if the cached function reference is not being used to prevent child re-renders or effect re-runs
  • C) useCallback breaks when used with event handlers
  • D) Functions inside useCallback cannot access state or props
▸ Show Answer

B — useCallback is an optimization, not a default. It caches a function reference so it stays the same between renders. This only matters when:

  • The function is passed as a prop to a child wrapped in React.memo (stable reference prevents child re-renders)
  • The function is in a useEffect dependency array (stable reference prevents effect re-runs)
  • The function is returned from a custom hook (consumers might use it in either scenario above)

If none of these apply, useCallback adds overhead (dependency comparison) with zero benefit. The component still re-renders — useCallback doesn't prevent that.

Question 2 — Code Analysis

A developer notices that their Shipping component re-renders every time the parent's counter changes, even though shipping data hasn't changed. They wrap Shipping in React.memo but it still re-renders. Why?

function Page() {
    const [count, setCount] = useState(0);

    const handleShippingChange = (option) => {
        console.log('Shipping changed to:', option);
    };

    return (
        <div>
            <button onClick={() => setCount(count + 1)}>Count: {count}</button>
            <Shipping onShippingChange={handleShippingChange} />
        </div>
    );
}

const Shipping = React.memo(({ onShippingChange }) => {
    return <select onChange={(e) => onShippingChange(e.target.value)}>...</select>;
});

a) Explain why React.memo alone does not prevent re-renders here. (2 pts)

b) Show the fix. (2 pts)

▸ Show Answer

a) Every time Page re-renders (when count changes), the handleShippingChange function is recreated as a new function object. Even though the function logic is identical, JavaScript sees a new reference. React.memo does a shallow comparison of props — since onShippingChange is a new reference every time, it looks like the prop changed, and Shipping re-renders.

b) Wrap handleShippingChange in useCallback:

const handleShippingChange = useCallback((option) => {
    console.log('Shipping changed to:', option);
}, []); // empty deps = stable reference forever

Now the function reference stays the same across re-renders. React.memo sees that the prop hasn't changed and skips re-rendering Shipping.

Question 3 — Code Analysis

Compare these two implementations of a "add todo" handler:

// Version A
const handleAddTodo = useCallback((text) => {
    setTodos([...todos, { id: nextId++, text }]);
}, [todos]);

// Version B
const handleAddTodo = useCallback((text) => {
    setTodos(prev => [...prev, { id: nextId++, text }]);
}, []);

Both work correctly, but Version B is considered the better approach. Explain why.

▸ Show Answer

Version A depends on the todos state variable, so todos must be in the dependency array. Every time todos changes (i.e., every time a todo is added), a new function is created. This defeats the purpose of useCallback — the function reference changes on every state update.

Version B uses the functional updater pattern (prev => ...). Instead of reading todos from the closure, it receives the previous state as a parameter. This means the callback does not reference todos at all, so the dependency array can be empty []. The function reference remains stable across all re-renders, which is the whole point of useCallback.

Rule: When a memoized callback needs to read current state, use the functional updater to remove state from the dependency array.

Question 4

<code>function Page() {
  const [count, setCount] = useState(0);

▸ Show Answer

(Slide ~435-438)

Without useCallback, every time Page re-renders (e.g., when count changes), a new handleShippingChange function object is created. Even though the function logic is identical, React sees it as a different reference. If Shipping is wrapped in React.memo, it would still re-render because its onShippingChange prop changed reference.

Question 5

The slides describe a "betri leid" (better way) for updating state from a memoized callback. What is the functional updater pattern, and why is it the better approach?

Explain the difference between these two approaches:

<code>// Approach A
const handleAddTodo = useCallback((text: string) => {
  setTodos([...todos, { id: nextId++, text }]);
}, [todos]);

// Approach B - "betri leid"
const handleAddTodo = useCallback((text: string) => {
  setTodos(prev => [...prev, { id: nextId++, text }]);
}, []);
</code>
▸ Show Answer

(Slide ~440-442)

The functional updater pattern (Approach B / "betri leid") is better because:

  • Approach A depends on the current todos state, so todos must be in the dependency array. Every time todos changes, a new function is created, defeating the purpose of useCallback.
  • Approach B uses the functional updater prev => [...] which receives the previous state as an argument. This means the callback does not need to reference todos at all, so the dependency array can be empty []. The function reference stays the same across all re-renders.

The "betri leid" removes the need for state variables in the dependency array by reading state through the updater function argument instead.

Question 6

The slides show an example of preventing useEffect from firing too often using useCallback. Explain how wrapping buildQueryParams in useCallback solves the problem.

<code>function SearchPage({ query, filters }: Props) {
  const buildQueryParams = useCallback(() => {
    return new URLSearchParams({
      q: query,
      ...filters,
    }).toString();
  }, [query, filters]);

  useEffect(() => {
    const params = buildQueryParams();
    fetch(`/api/search?${params}`);
  }, [buildQueryParams]);
}
</code>
▸ Show Answer

(Slide ~445-448)

Without useCallback, buildQueryParams is recreated on every render. Since it is in the useEffect dependency array, the effect runs on every render — even when query and filters haven't changed.

By wrapping buildQueryParams in useCallback with [query, filters] as dependencies, the function reference only changes when query or filters actually change. This means the useEffect only fires when the search parameters truly change, preventing unnecessary API calls.

Question 7

According to the slides, when should you wrap functions in useCallback when building custom hooks? Why?

▸ Show Answer

(Slide ~450-452)

When building custom hooks, you should wrap any returned functions in useCallback. This is because you don't know how the consumer of your hook will use the function — they might pass it as a prop to a memoized component or include it in a useEffect dependency array. Wrapping it in useCallback ensures that consumers don't get unnecessary re-renders or effect re-runs. This is part of "optimizing custom hooks."

SECTION B: useMemo (Slides 455-480)

Question 8 — Multiple Choice

A component has an expensive .filter() and .reduce() on a large array of products. The component re-renders frequently because of a chat widget in the sidebar updating state. The filtering and reducing produce the same result every time because the products haven't changed. What is the best optimization?

  • A) Move the filter/reduce logic into a useEffect and store the result in state
  • B) Wrap the calculation in useMemo with the products array as a dependency
  • C) Wrap the calculation in useCallback so it doesn't run on every render
  • D) Use useRef to cache the calculation result
▸ Show Answer

B — useMemo caches the RESULT of a calculation. With [products] as the dependency, the expensive filter/reduce only re-runs when products actually changes. On re-renders caused by the chat widget, the cached result is returned instantly.

Why the others are wrong:

  • A — Using useEffect to store derived data in state is an anti-pattern. It adds unnecessary complexity (effect + state), causes an extra render (effect runs after render), and the data is already derivable from props.
  • C — useCallback caches a function reference, not a computed value. The calculation would still run every time you call the function.
  • D — useRef persists values but has no dependency-checking mechanism. You'd have to manually check if products changed, which is what useMemo does automatically.

Question 9 — Code Analysis

A developer writes:

function Dashboard({ data }) {
    const sorted = data.sort((a, b) => b.value - a.value);
    const top5 = sorted.slice(0, 5);

    return <TopItems items={top5} />;
}

This component re-renders 10 times per second because of a live clock in the parent. Identify two problems and show how to fix them.

▸ Show Answer

Problem 1: Expensive recalculation on every render. The .sort() and .slice() run on every render (10 times per second), even when data hasn't changed. Fix: wrap in useMemo.

Problem 2: .sort() mutates the original array. Array.sort() sorts in place — it modifies the data prop directly. This violates React's rule that props should not be mutated. Fix: create a copy first with [...data].

function Dashboard({ data }) {
    const top5 = useMemo(() => {
        return [...data]                            // copy to avoid mutation
            .sort((a, b) => b.value - a.value)     // sort the copy
            .slice(0, 5);
    }, [data]);                                     // only recalculate when data changes

    return <TopItems items={top5} />;
}

Now the sort/slice only runs when data actually changes, and the original array is never mutated.

Question 10

Look at the following ProductDashboard example from the slides. Why is useMemo used here, and what problem does it solve?

<code>function ProductDashboard({ products, category }: Props) {
  const filteredProducts = useMemo(() => {
    return products.filter(p => p.category === category);
  }, [products, category]);

  const stats = useMemo(() => {
    return {
      total: filteredProducts.length,
      avgPrice: filteredProducts.reduce((sum, p) => sum + p.price, 0)
        / filteredProducts.length,
    };
  }, [filteredProducts]);

  return (
    <div>
      <h2>Stats: {stats.total} products, avg ${stats.avgPrice.toFixed(2)}</h2>
      {filteredProducts.map(p => (
        <ProductCard key={p.id} product={p} />
      ))}
    </div>
  );
}
</code>
▸ Show Answer

(Slide ~462-466)

useMemo is used here to skip expensive recalculations:

  1. filteredProducts uses useMemo with [products, category] — the filtering only re-runs when the products array or category changes. If the parent re-renders for other reasons (e.g., unrelated state change), the expensive .filter() operation is skipped and the cached result is returned.
  2. stats uses useMemo with [filteredProducts] — the statistics calculation (total count, average price with .reduce()) only re-runs when filteredProducts changes.

Without useMemo, both the filtering and the statistics calculation would run on every single re-render, which is wasteful when products and category haven't changed.

Question 11

What is the difference between useMemo and useCallback? When would you use one over the other?

▸ Show Answer

(Slides ~455, ~430)

  • useMemo caches the result of a calculation (a value).
  • useCallback caches a function definition (a function reference).

useCallback(fn, deps) is essentially equivalent to useMemo(() => fn, deps).

Use useMemo when you have an expensive computation whose result you want to cache. Use useCallback when you want to cache a function reference to prevent unnecessary re-renders of child components or unnecessary effect re-runs.

Question 12 — Written Answer (4 pts)

For each scenario below, which hook should you use: useState, useRef, useMemo, or useCallback? Justify each choice in one sentence.

a) You need to store a WebSocket connection instance that persists across renders.

b) You need to pass an onDelete handler to a memoized child component.

c) You need to toggle a modal open/closed and the UI should update when it changes.

d) You need to compute the total price from a list of cart items, and the calculation should not re-run unless the cart changes.

▸ Show Answer

a) useRef — a WebSocket connection does not affect rendering (changing it should not cause a re-render), and it needs to persist across renders. useRef stores a mutable value without triggering re-renders.

b) useCallback — the onDelete function reference must stay stable so the memoized child (React.memo) sees the same prop and skips re-rendering. useCallback caches the function reference.

c) useState — the modal open/closed state directly affects what the UI renders (show/hide the modal). useState triggers a re-render when the value changes, which is exactly what you need.

d) useMemo — the total price is a derived value computed from cart items. useMemo caches the result and only recalculates when the dependency (cart) changes, avoiding the expensive computation on unrelated re-renders.

SECTION C: Forms, Events & Refs (Slides 620-640)

Question 13 — Written Answer (4 pts)

You are building a form with 15 input fields. A colleague suggests managing all 15 fields with separate useState hooks. Another colleague suggests using react-hook-form.

a) What performance problem would occur with 15 separate useState hooks? (2 pts)

b) How does react-hook-form avoid this problem? (2 pts)

▸ Show Answer

a) With 15 separate useState hooks, each input would need a controlled approach: value={fieldState} and onChange={(e) => setFieldState(e.target.value)}. Every keystroke in ANY field calls a setter, which triggers a re-render of the ENTIRE component. With 15 fields, even typing one character re-renders all 15 inputs. This creates noticeable lag, especially on slower devices.

b) react-hook-form uses uncontrolled inputs by default. When you call register("fieldName"), it attaches refs to the inputs — the DOM manages the input values, not React state. The component does NOT re-render on every keystroke. It only re-renders when validation triggers (based on the mode setting: onSubmit, onBlur, or onChange). On submit, react-hook-form reads all values from the DOM at once via refs.

Question 14

What is a SyntheticEvent in React? List at least four event handlers mentioned in the slides.

▸ Show Answer

(Slide ~622-624)

A SyntheticEvent is React's abstraction of the native DOM event object. It wraps the browser's native event to provide a consistent API across all browsers.

Event handlers from the slides include:

  • onClick — fires when an element is clicked
  • onContextMenu — fires on right-click
  • onMouseMove — fires when the mouse moves over an element
  • onSubmit — fires when a form is submitted
  • onChange — fires when the value of an input changes

Question 15

What is the definition of useRef from the slides? How does it differ from useState when it comes to re-renders?

▸ Show Answer

(Slide ~628-630)

useRef according to the slides: "Changes made to the reference will not trigger a re-render and the reference can be mutated."

Key difference from useState:

  • useState — changing the value triggers a re-render, the value is immutable (you replace it via the setter function).
  • useRef — changing .current does not trigger a re-render, and the reference can be directly mutated.

Question 16

Compare the two approaches to reading text input values from the slides:

Approach A — Using a ref:

<code>function Form() {
  const inputRef = useRef<HTMLInputElement>(null);

  function handleSubmit(e: React.FormEvent) {
    e.preventDefault();
    const value = inputRef.current?.value;
    console.log('Submitted:', value);
  }

  return (
    <form onSubmit={handleSubmit}>
      <input ref={inputRef} type="text" />
      <button type="submit">Submit</button>
    </form>
  );
}
</code>

Approach B — Controlled input:

<code>function Form() {
  const [value, setValue] = useState('');

  function handleSubmit(e: React.FormEvent) {
    e.preventDefault();
    console.log('Submitted:', value);
  }

  return (
    <form onSubmit={handleSubmit}>
      <input
        type="text"
        value={value}
        onChange={(e) => setValue(e.target.value)}
      />
      <button type="submit">Submit</button>
    </form>
  );
}
</code>

When should you use each approach?

▸ Show Answer

(Slide ~630-634)

Approach A (ref / uncontrolled input):

  • The DOM owns the input value. React reads it only when needed (e.g., on submit).
  • No re-render happens as the user types.
  • Best for: simple forms where you only need the value on submit, or when integrating with non-React code.

Approach B (controlled input):

  • React state owns the input value via value and onChange.
  • Every keystroke triggers a re-render because setValue is called.
  • Best for: when you need to validate, transform, or react to input changes in real time (e.g., showing character count, live search, conditional rendering based on input).

The slides show that refs are the simpler approach when you just need to "grab" a value, while controlled inputs give you full control over the input at the cost of more re-renders.

Question 17

According to the slides, what does forwardRef do? Why is it described as something that "encapsulates the code"?

▸ Show Answer

(Slide ~635-636)

forwardRef "encapsulates the code" for passing refs from a parent component to a child component's DOM element. By default, React components do not expose their internal DOM nodes. forwardRef wraps a component so that the parent can pass a ref that gets attached to a specific DOM element inside the child.

Example:

<code>const FancyInput = forwardRef<HTMLInputElement, Props>((props, ref) => {
  return <input ref={ref} className="fancy" {...props} />;
});

// Parent can now do:
const inputRef = useRef<HTMLInputElement>(null);
<FancyInput ref={inputRef} />
</code>

This encapsulation means the parent doesn't need to know the internal structure of the child — it just gets access to the ref.

Question 18

What does useImperativeHandle do according to the slides? Give an example of when you would use it.

▸ Show Answer

(Slide ~637-638)

useImperativeHandle lets you "customize the handle exposed as a ref." Instead of exposing the full DOM node, you can expose a custom object with only specific methods.

Example use case: You have a custom input component and want the parent to only be able to call .focus() and .scrollIntoView(), but not access any other DOM properties:

<code>const CustomInput = forwardRef<{ focus: () => void }, Props>((props, ref) => {
  const inputRef = useRef<HTMLInputElement>(null);

  useImperativeHandle(ref, () => ({
    focus() {
      inputRef.current?.focus();
    },
  }));

  return <input ref={inputRef} {...props} />;
});
</code>

This gives you control over what the parent component can do with the ref.

Question 20

Look at this TodoList code. Identify where useRef, controlled inputs, and event handling are used. What would happen if you replaced the useRef with useState for the input?

<code>function TodoList() {
  const [todos, setTodos] = useState<string[]>([]);
  const inputRef = useRef<HTMLInputElement>(null);

  function handleAdd() {
    const text = inputRef.current?.value.trim();
    if (text) {
      setTodos(prev => [...prev, text]);
      inputRef.current!.value = '';
      inputRef.current!.focus();
    }
  }

  return (
    <div>
      <input ref={inputRef} placeholder="New todo" />
      <button onClick={handleAdd}>Add</button>
      <ul>
        {todos.map((t, i) => <li key={i}>{t}</li>)}
      </ul>
    </div>
  );
}
</code>
▸ Show Answer

(Slide ~632-634)

In this TodoList code:

  • useRef: inputRef is used to create an uncontrolled input. The DOM manages the input value. We read inputRef.current?.value on button click and clear it with inputRef.current!.value = ''. We also call inputRef.current!.focus() to refocus after adding.
  • useState (controlled state): todos is managed as React state because it directly affects rendering (the list of <li> elements).
  • Event handling: handleAdd is an event handler attached via onClick on the button.

If you replaced useRef with useState for the input:

  • Every keystroke would trigger a re-render (because setValue calls cause re-renders).
  • The input would become a "controlled" input requiring value={inputValue} and onChange={(e) => setInputValue(e.target.value)}.
  • The component would re-render more frequently, but you would gain real-time access to the input value for features like validation or live filtering.
  • The current approach with useRef is simpler and avoids unnecessary re-renders since the input value doesn't affect the rendered output until the user clicks "Add."
