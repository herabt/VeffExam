Question 4 — Written Answer

Explain the difference between a parallel route and an intercepting route in Next.js. Give a concrete use-case for each.

▸ Show Answer

Parallel routes (@slot folders) allow multiple pages to be rendered simultaneously in the same layout. They are defined with the @folderName convention and are passed as props to the parent layout. Example use-case: a dashboard with @analytics and @team panels side-by-side that can load and navigate independently.

Intercepting routes ((.)folder, (..)folder, etc.) let you render a route in the context of the current layout, intercepting navigation. Classic use-case: clicking a photo in a feed shows it in a modal (intercepted route) while a direct URL or page-refresh shows it on its own dedicated page (the real route).

Folder prefixes: (.) same level, (..) one level up, (...) from root.

Question 9 — Written Answer

A team-mate imports a Server Component inside a Client Component and gets a build error. Explain why this happens and describe two ways to resolve it.

▸ Show Answer

Why: When a Server Component is imported inside a "use client" module, it is pulled across the client boundary and becomes part of the client bundle. Server Components can contain server-only code (Node APIs, secrets, async DB calls) that cannot run in the browser, so Next.js throws an error.

Resolution 1 — Pass as children prop: Keep the Server Component in the parent (server) layout and pass it into the Client Component as children or another prop. The Server Component stays on the server; only its rendered output is sent to the client.

Resolution 2 — Move the Server Component higher: Restructure so the Client Component is a leaf and the Server Component wraps or sits alongside it rather than being imported by it.

Question 16 — Written Answer

Explain how you would use <Suspense> manually to stream parts of a page independently, without relying on loading.tsx. Write a short example.

▸ Show Answer

Wrap any async Server Component in <Suspense fallback={...}>. The fallback renders instantly while the async component resolves on the server and is then streamed in.

// app/dashboard/page.tsx (Server Component)
import { Suspense } from 'react'
import RevenueChart from './RevenueChart'
import RecentOrders from './RecentOrders'

export default function DashboardPage() {
  return (
    <main>
      <h1>Dashboard</h1>
      <Suspense fallback={<p>Loading chart...</p>}>
        <RevenueChart />
      </Suspense>
      <Suspense fallback={<p>Loading orders...</p>}>
        <RecentOrders />
      </Suspense>
    </main>
  )
}

Each component fetches its own data in parallel. Whichever resolves first is streamed to the client first, reducing time-to-interactive.

Question 18 — Written Answer

Explain what useActionState (formerly useFormState) and useFormStatus do in the context of Server Actions. What are the rules about where each can be used?

▸ Show Answer

useActionState(action, initialState) — a React hook that wraps a Server Action. It returns [state, formAction, isPending]. state holds the value returned by the last action invocation. Question 20 — Written Answer

What is Next.js Middleware and where does the middleware.ts file live? Describe two practical use-cases and explain what NextResponse.redirect() vs NextResponse.rewrite() do.

▸ Show Answer

Location: middleware.ts goes at the root of the project (same level as app/ or pages/), not inside app/.

What it does: Runs on the Edge runtime before a request reaches the route handler/page, allowing inspection and transformation of requests and responses.

Use-cases:

  • Authentication — check a session cookie; redirect unauthenticated users to /login.
  • Internationalisation — detect the user's locale from Accept-Language header and rewrite the URL to /en/....

Question 23 — Written Answer

Explain how Redux Toolkit uses Immer inside createSlice reducers. Why is it safe to write state.value += 1 in a reducer even though Redux requires immutable updates?

▸ Show Answer

RTK integrates Immer into every reducer created by createSlice (and createReducer). Immer wraps the state in a Proxy object. When you mutate the proxy (e.g., state.value += 1), Immer tracks all changes and produces a new immutable state snapshot behind the scenes — the original state object is never actually mutated.

This means you can write straightforward "mutable-style" code in your reducers while Redux still receives a brand-new state reference, triggering proper re-renders. You must either mutate the proxy or return a new value — not both.

Explanation: createAsyncThunk('slice/action', async (arg) => ...) generates three action creators: action.pending, action.fulfilled, and action.rejected. These are dispatched automatically at the start, on success, and on error respectively. You handle them in extraReducers inside createSlice to update loading state, data, and error fields.

Question 26 — Written Answer

Describe the unidirectional data flow in Redux. What are the four steps from a user interaction to a UI update?

▸ Show Answer
  1. Event / User interaction — e.g., a button click in the UI triggers a call to dispatch(action).
  2. Dispatch — the action (a plain object with a type field and optional payload) is sent to the Redux store.
  3. Reducer — the store calls the root reducer with (currentState, action). The reducer is a pure function; it returns a new state without mutating the old one.
  4. Store update + re-render — the store saves the new state. React components subscribed via useSelector receive the new state and re-render with the updated data.

SECTION F: Context API & Zustand

Question 27 — Multiple Choice

What is the main performance pitfall when using React Context for frequently-changing state?

  • A) Context values are stored in the DOM and reading them is slow
  • B) Every consumer of the context re-renders whenever the context value changes, even if it only uses part of the value
  • C) Context does not work with concurrent React features
  • D) Context values cannot hold objects, only primitives
▸ Show Answer

Answer: B

Explanation: React Context has no built-in selector mechanism. Any time the context value object reference changes, all consumers re-render — even those that only use an unrelated slice of the value. This makes Context unsuitable for high-frequency state (e.g., mouse position, real-time data). It is best suited for low-frequency, globally-shared state like theme or authentication.

Question 28 — Written Answer

Compare the Zustand useStore selector pattern to naive Context usage. Why do Zustand selectors prevent unnecessary re-renders?

▸ Show Answer

Zustand's useStore(selector) hook subscribes a component only to the slice of state returned by the selector function. Zustand performs a shallow (or strict equality) comparison between the previous and next selector result. If the result is the same, the component is not re-rendered, even if other parts of the store changed.

// Only re-renders when bears count changes
const bears = useStore((state) => state.bears)

// vs Context: re-renders whenever ANY value in the provider changes

Context consumers re-render on every provider value change because React only does a reference equality check on the entire context value. Zustand solves this with per-component, fine-grained subscriptions — similar to Redux's useSelector.

Question 34 — Written Answer

When should you use the Controller component from react-hook-form, and how does it work? Give an example with a UI library input.

▸ Show Answer

When to use: Controller is needed for controlled third-party components that do not expose a native ref (e.g., custom dropdowns, date-pickers from MUI, Radix UI selects, or React-Select). These components require value and onChange props rather than a DOM ref.

Controller acts as a bridge: it wraps the component, subscribes to RHF's store, and passes field.value, field.onChange, and field.onBlur to the inner component.

import { useForm, Controller } from 'react-hook-form'
import Select from 'react-select'

function MyForm() {
  const { control, handleSubmit } = useForm()

  return (
    <form onSubmit={handleSubmit(console.log)}>
      <Controller
        name="country"
        control={control}
        rules={{ required: true }}
        render={({ field }) => (
          <Select
            {...field}
            options={[{ value: 'is', label: 'Iceland' }]}
          />
        )}
      />
    </form>
  )
}

Question 37 — Written Answer

Explain the difference between watch and getValues in react-hook-form. When should you use each?

▸ Show Answer

watch(fieldName) — subscribes the component to re-render whenever the watched field changes. Use it when the UI needs to react to a field value in real-time, e.g., showing a conditional section when a checkbox is ticked or computing a live preview.

getValues(fieldName?) — reads the current value(s) from the form store without subscribing to updates and without triggering a re-render. Use it inside event handlers or effects where you just need to read the value once (e.g., "copy shipping to billing" button, validating one field relative to another in a validate function).

Rule of thumb: if you need the value in JSX / render output → watch. If you need it in a callback → getValues.

Question 39 — Written Answer

You have a form with a field named email. Describe how to display a custom error message for (a) a required error, (b) a pattern error, and (c) a server-side error returned after submission.

Question 41 — Written Answer

You are building a Next.js App Router application. Describe a complete architecture decision process: when would you reach for (a) Server Component data fetching, (b) RTK Query, (c) Zustand, and (d) react-hook-form, and how can they coexist?

▸ Show Answer

(a) Server Component data fetching — for any data needed to render the initial page that does not require user interaction or real-time updates (product listings, blog posts, user profile on a settings page). Fetch directly in the Server Component; no client-side state library needed.

(b) RTK Query — for data that must be fetched and mutated from the client side, especially when multiple components need the same cached data (e.g., a shopping cart, notifications badge, or any CRUD UI). RTK Query handles loading/error/cache states automatically.

(c) Zustand — for lightweight client-side UI state that doesn't map directly to server data: modal open/close, sidebar state, wizard step, or cross-component ephemeral state that doesn't warrant the overhead of Redux.

(d) react-hook-form — for all form state: field values, validation errors, dirty/touched tracking, submission state. RHF keeps form state out of React state and out of global stores — it is self-contained per form instance.

Coexistence example: A "Create Order" page uses a Server Component to fetch product options. A Client Component renders the form (RHF). On submit, the form calls an RTK Query mutation to POST the order. On success, Zustand updates a global notification store.

Question 43 — Written Answer

A colleague adds "use client" to every component "just to be safe". Explain the performance implications and describe the correct strategy for deciding where to place the client boundary.

▸ Show Answer

Performance implications:

  • Every "use client" component (and everything it imports) ships to the browser as JavaScript, inflating the client bundle.
  • Server Components have zero JavaScript cost on the client — they render on the server and send HTML. Making everything a Client Component eliminates this advantage.
  • Server-side data fetching, direct DB access, and secret keys are also lost — they cannot live in Client Components.

Correct strategy — push the boundary to the leaves:

  1. Start with Server Components by default.
  2. Only add "use client" to the smallest component that actually needs interactivity (hooks, event handlers, browser APIs).
  3. Keep data-fetching and layout at the server level; pass data down as props to small client leaves.
  4. Or use the children composition pattern to thread Server Component output through a Client Component wrapper.

Question 45 — Written Answer

Explain how to implement optimistic UI updates using RTK Query's onQueryStarted lifecycle. Walk through the rollback pattern if the server request fails.

▸ Show Answer

RTK Query mutations expose onQueryStarted(arg, { dispatch, queryFulfilled, getState }) in the endpoint definition. You can patch the cache optimistically before the request completes:

updatePost: builder.mutation({
  query: (patch) => ({ url: `/posts/${patch.id}`, method: 'PATCH', body: patch }),
  async onQueryStarted({ id, ...patch }, { dispatch, queryFulfilled }) {
    // 1. Optimistically update the cache
    const patchResult = dispatch(
      postsApi.util.updateQueryData('getPosts', undefined, (draft) => {
        const post = draft.find((p) => p.id === id)
        if (post) Object.assign(post, patch)
      })
    )
    try {
      // 2. Wait for the real response
      await queryFulfilled
    } catch {
      // 3. Roll back on failure
      patchResult.undo()
    }
  },
})

The updateQueryData utility (powered by Immer) creates an optimistic patch and returns an undo function. If queryFulfilled rejects, calling patchResult.undo() reverses the optimistic change, restoring the previous cache state.

Question 47 — Written Answer

Describe the complete flow of a form submission using a Next.js Server Action without any client-side JavaScript (progressive enhancement). What must be true for this to work?

▸ Show Answer

Flow:

  1. The form has a native HTML <form action={serverAction}>. No onSubmit JS handler needed.
  2. When the user submits, the browser serialises the form fields as multipart/form-data and POSTs to the Next.js Server Action endpoint.
  3. Next.js invokes the Server Action with a FormData object on the server.
  4. The action reads data via formData.get('fieldName'), validates, and performs the mutation.
  5. The server re-renders the page with updated state (or calls redirect()).

Requirements for progressive enhancement:

  • Use a real <form> element with a native submit button (<button type="submit">).
  • Pass the Server Action to the action attribute, not onSubmit.
  • The action must be defined with "use server".
  • All form fields must have name attributes so they appear in FormData.

When JS is available, React intercepts the submit and calls the action without a full page reload (enhanced behaviour). Without JS, the form still works via standard browser form submission.

Question 48 — Multiple Choice

You have a createSlice with a reducer that returns a new state object instead of mutating the Immer draft. Which of the following is true?

  • A) This is forbidden — you must always mutate the draft
  • B) This is valid — you can either mutate the draft OR return a new value, but not both
  • C) Returning a new value from a reducer causes a TypeScript error in RTK
  • D) Immer ignores returned values and always uses the mutated draft
▸ Show Answer

Answer: B

Explanation: Immer supports two patterns in a reducer: (1) mutate the proxy draft directly and return nothing, or (2) return a completely new value (bypassing Immer). You cannot do both — if you mutate the draft AND return a new value, Immer throws an error. Returning a new value is useful when you want to replace the entire state slice (e.g., resetting to initial state).

// Pattern 1 — mutate draft (Immer handles immutability)
increment(state) { state.count += 1 }

// Pattern 2 — return new value
reset() { return initialState }

// INVALID — do not do both
broken(state) { state.count += 1; return { count: 0 } }

Question 49 — Written Answer

What is the difference between generateStaticParams and generateMetadata in Next.js App Router? Can they both be exported from the same file?

▸ Show Answer

generateStaticParams — exported from a dynamic page.tsx (e.g., app/blog/[slug]/page.tsx). Returns an array of params objects that Next.js should pre-render as static pages at build time. Replaces getStaticPaths from the Pages Router.

generateMetadata — exported from any page.tsx or layout.tsx. Returns a Metadata object (title, description, OpenGraph, etc.) optionally after fetching data. Replaces the static metadata export when the metadata depends on route params or remote data. Replaces getServerSideProps-based head injection from the Pages Router.

Can they coexist? Yes. Both can be exported from the same page.tsx file. Next.js deduplicates any shared fetch calls between them automatically using the built-in request memoisation (React's cache).
