SECTION A: BEM (Blocks, Elements, Modifiers)

Question 1 — Code Analysis

A developer writes the following CSS for a navigation component:

#main-nav { }
#main-nav .item { }
#main-nav .item.active { }
#main-nav .item .icon { }

a) Identify two problems with this approach from a BEM perspective. (2 pts)

b) Rewrite these selectors using proper BEM naming conventions. (2 pts)

▸ Show Answer

a) Two problems:

  • Using an ID selector (#main-nav) — BEM requires class selectors only. IDs create high specificity, making styles hard to override and components impossible to reuse (IDs must be unique per page).
  • Nested/dependent selectors (#main-nav .item) — BEM blocks should have no dependency on other blocks or elements on the page. The styling is tightly coupled to the DOM structure.

b) Proper BEM:

.nav { }                /* Block */
.nav__item { }          /* Element */
.nav__item--active { }  /* Modifier */
.nav__icon { }          /* Element */

Each selector is a flat class with no nesting. The block (nav) is standalone, elements use __, modifiers use --. This makes the styles independent of DOM structure and easy to reuse.

Question 5 — Written Answer (4 pts)

You are building a product card component. The card has a title, an image, a price, and a "Buy" button. The card can be in a "featured" state (larger, highlighted border) and the button can be "disabled" when the product is out of stock.

a) Write the HTML for this card using proper BEM class names. (2 pts)

b) Explain how the modifier approach is better than creating a completely separate class like .featured-card. (2 pts)

▸ Show Answer

a) HTML with BEM:

<div class="card card--featured">
  <img class="card__image" src="..." alt="..." />
  <h3 class="card__title">Product Name</h3>
  <span class="card__price">$29.99</span>
  <button class="card__button card__button--disabled">Out of Stock</button>
</div>

b) The modifier approach (.card--featured) is better because:

  • The featured card shares all base styles from .card — the modifier only adds or overrides what is different (e.g., larger size, border color). No code duplication.
  • A separate class .featured-card would need to duplicate all the base card styles, creating maintenance problems: every change to the base card must be applied in two places.
  • BEM modifiers are always applied alongside the base class (class="card card--featured"), making it clear that a featured card IS a card with a variation, not a different component.

SECTION B: CSS Variables / Custom Properties

Question 6 — Written Answer (4 pts)

A website uses the color #63B4D1 in 47 different places across 12 CSS files. A designer asks you to change the brand color to #FF6B35.

a) Without CSS variables, what would you need to do? What could go wrong? (2 pts)

b) Show how CSS variables solve this problem. Write the CSS to define the variable and use it. (2 pts)

▸ Show Answer

a) Without CSS variables, you would need to find-and-replace #63B4D1 in all 47 locations across 12 files. What could go wrong: you might miss some instances (especially if some use rgb() or uppercase hex), accidentally replace a different color that happens to be similar, or introduce typos. The color value is duplicated everywhere with no single source of truth.

b) With CSS variables, define the color once and reference it everywhere:

:root {
    --brand-color: #FF6B35;   /* change here = changes everywhere */
}

.nav { background-color: var(--brand-color); }
.button { border-color: var(--brand-color); }
.link { color: var(--brand-color); }

Now changing the brand color requires editing one line. CSS variables are defined with -- prefix and accessed with var(). They can also be overridden within specific selectors and updated dynamically with JavaScript.

SECTION C: Tailwind CSS

Question 11 — Written Answer (4 pts)

A colleague says: "Tailwind CSS and BEM both solve the same problem, so you only need to know one."

a) What problem do they share, and in what way do their solutions differ? (2 pts)

b) Name one concrete benefit of Tailwind's utility-first approach that BEM does not provide. (2 pts)

▸ Show Answer

a) Shared problem: Both address the challenge of organizing CSS and avoiding style conflicts. BEM solves this through a strict naming convention (.block__element--modifier) that avoids specificity wars by keeping selectors flat. Tailwind solves it by eliminating custom class names entirely — you apply pre-defined utility classes directly in your HTML (flex pt-4 text-center).

b) Tailwind benefit BEM doesn't provide: When you delete a component, its styles automatically disappear because they are in the HTML itself. With BEM, you have separate CSS files where unused classes can accumulate as dead code. Tailwind also eliminates "naming fatigue" — you never need to invent class names like .card__header--highlighted.

SECTION D: JavaScript Objects

Question 14 — Code Analysis (4 pts)

Study this code:

var original = { name: 'Alice', scores: [90, 85] };
var copy = original;

copy.name = 'Bob';
copy.scores.push(95);

console.log(original.name);    // Line A
console.log(original.scores);  // Line B

a) What will Lines A and B output? (2 pts)

b) Explain WHY changing copy also changed original. What does "manipulated by reference" mean in this context? (2 pts)

▸ Show Answer

a)

  • Line A: 'Bob'
  • Line B: [90, 85, 95]

b) Objects in JavaScript are mutable and manipulated by reference. When you write var copy = original, you are NOT creating a new object — you are creating a second variable that points to the SAME object in memory. Both original and copy hold a reference (like a pointer) to the same data. Any change through either variable affects the same object.

To create an independent copy, you would need: var copy = { ...original, scores: [...original.scores] } (shallow copy with spread, plus a copy of the nested array).

Question 15 — Code Analysis

Study this code:

var obj = {};
Object.defineProperty(obj, 'id', {
    value: 42,
    writable: false,
    enumerable: false,
    configurable: false
});

obj.id = 99;                    // Line A
console.log(obj.id);            // Line B
console.log(Object.keys(obj));  // Line C

a) What will Lines B and C output? (2 pts)

b) Explain why setting writable: false and enumerable: false produces this behavior. When would you want to use these attributes? (2 pts)

▸ Show Answer

a)

  • Line B: 42 — the assignment on Line A silently fails because writable: false prevents the value from being changed.
  • Line C: [] — Object.keys() only returns enumerable properties. Since enumerable: false, the id property is invisible to enumeration.

b)

  • writable: false makes the property read-only — any attempt to assign a new value is silently ignored (or throws in strict mode). Use this when you want a constant property that cannot be accidentally overwritten, like an internal ID.
  • enumerable: false hides the property from for...in loops, Object.keys(), and JSON.stringify(). This is why built-in methods like toString don't show up in for...in loops — they are defined as non-enumerable on Object.prototype.

Question 20 — Written Answer (4 pts)

Study this code:

function Animal(name) {
    this.name = name;
    this.speak = function() { return this.name + ' speaks'; };
}

function AnimalV2(name) {
    this.name = name;
}
AnimalV2.prototype.speak = function() { return this.name + ' speaks'; };

var a1 = new Animal('Cat');
var a2 = new Animal('Dog');

var b1 = new AnimalV2('Cat');
var b2 = new AnimalV2('Dog');

Both approaches produce objects that behave the same way. However, one approach is significantly better for memory efficiency.

a) Which approach is more memory-efficient, and why? (2 pts)

b) Explain what happens when you call b1.speak() — how does JavaScript find the speak method on b1 even though it's not defined directly on the object? (2 pts)

▸ Show Answer

a) AnimalV2 (prototype approach) is more memory-efficient. In the first approach, this.speak = function() {...} creates a new function object for every instance. If you create 1000 animals, you have 1000 identical copies of the speak function in memory. In the prototype approach, speak is defined once on AnimalV2.prototype and shared by all instances — 1000 animals still have only 1 copy of the function.

b) When you call b1.speak(), JavaScript uses the prototype chain:

  1. It first looks for speak directly on b1 — it's not there (b1 only has name).
  2. It then looks on b1's prototype, which is AnimalV2.prototype — it finds speak there and calls it.
  3. If it wasn't there either, it would continue up the chain to Object.prototype.

This lookup is called the prototype chain. Every object has a link to its prototype, and JavaScript traverses this chain automatically when accessing properties.
