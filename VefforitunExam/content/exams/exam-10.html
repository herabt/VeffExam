Question 10 — Multiple Choice

Given the following slide example about structural type system, which call will FAIL and why?

interface Point {
  x: number;
  y: number;
}

function logPoint(p: Point) { console.log(`${p.x}, ${p.y}`); }

const point3d = { x: 1, y: 2, z: 3 };
logPoint(point3d); // logs "1, 2"

const rect = { x: 1, y: 2, width: 30, height: 40 };
logPoint(rect); // logs "1, 2"

const animal = { type: 'cow', name: 'Fred' };
logPoint(animal); // fails
  • A) logPoint(point3d) fails because it has an extra z property
  • B) logPoint(rect) fails because it has width and height properties
  • C) logPoint(animal) fails because animal does not have x and y properties -- the shape-matching only requires a subset of the object's fields to match, but the object must declare the required properties
  • D) All three calls succeed
▸ Show Answer

Answer: C

Explanation: logPoint(animal) fails because animal has { type, name } but no x or y properties. The slides explain that shape-matching only requires a subset of the object's fields to match, so point3d (has x, y, z) and rect (has x, y, width, height) both pass because they include x and y. But animal does not have the required x and y.

Question 16 — Scenario: Migrating to TypeScript

You are joining a team that has a large existing JavaScript codebase (~200 files). The team wants to start using TypeScript. Based on what the slides cover about TypeScript:

a) The team lead wants to convert all 200 files to TypeScript at once before deploying. Another developer suggests converting files one at a time. Which approach aligns with how TypeScript is designed to be adopted, and why?

b) After converting a file to TypeScript, the developer runs tsc and sees 15 type errors. They are concerned that no .js output file was generated. Is this concern valid? Explain what actually happens.

c) A junior developer asks: "If TypeScript types are removed during compilation, what's the point? The browser never sees them." Explain the value TypeScript provides despite types being stripped out.

▸ Show Answer

a) Converting files one at a time is the correct approach. The slides state: "TypeScript is built to integrate gradually to an existing JavaScript application, by replacing chunks at a time from VanillaJS to TypeScript." A big-bang conversion of 200 files would be risky and unnecessary — TypeScript is explicitly designed for gradual adoption.

b) The concern is NOT valid. The slides explain: "Although TypeScript generates errors, we still see an output in our generated .js files." TypeScript still generates the .js output even when there are type errors, precisely because it is "built with migration in mind." The errors are warnings about potential issues, not blockers that prevent compilation. (Note: the --noEmitOnError flag can change this behavior, but the default allows output with errors.)

c) The value of TypeScript is in the development experience, not runtime behavior. Types provide: (1) Error detection before execution — you catch bugs while writing code, not when users encounter crashes. (2) Auto-complete and editor intelligence — the slides state that explicit types let "our code editor provide us with auto-complete features and suggestions." (3) Self-documenting code — interfaces and type annotations make it clear what a function expects and returns, reducing the need to read implementation details. The types serve their purpose during development and are safely removed for production.

Question 17 — Code Analysis: Type Error

Study the following TypeScript code. What error(s) will TypeScript report, and why? What is the correct fix?

interface Product {
  name: string;
  price: number;
  inStock: boolean;
}

function displayProduct(product: Product) {
  console.log(`${product.name}: $${product.price}`);
}

const shoes = { name: "Running Shoes", price: 89.99 };
displayProduct(shoes);
▸ Show Answer

Error: TypeScript reports that shoes is missing the required property inStock from the Product interface. The shoes object has { name: string, price: number } but the Product interface requires { name: string, price: number, inStock: boolean }.

Why: TypeScript uses a structural type system — it checks that the object has at least all the required properties. Having extra properties is fine (that's why point3d with {x, y, z} works for a Point {x, y} interface), but MISSING required properties is an error. The shape must contain at minimum all fields defined in the interface.

Fix — choose one:

  • Add the missing property: const shoes = { name: "Running Shoes", price: 89.99, inStock: true };
  • Make the property optional in the interface: inStock?: boolean;
  • Use Partial<Product> as the parameter type if all fields should be optional
