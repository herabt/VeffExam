Practice Exam 1 — Flexbox & CSS Grid

SECTION A: FLEXBOX FUNDAMENTALS

Question 1 — Fill in the blanks (4 pts)

Complete the following sentences about Flexbox. Each blank is a CSS property name, value, or keyword.

When you set display: flex on a container, its direct children become flex items and are laid out along the 1. ________ axis by default (left to right). The shorthand property 2. ________ lets you combine flex-direction and flex-wrap in a single declaration. If a flex item should override the container's align-items value for itself alone, you use the 3. ________ property on that item. By default, flex items will not wrap to a new line because flex-wrap defaults to 4. ________.

▸ Show Answer

1. main

Why: Flexbox always lays items along a main axis and a perpendicular cross axis. The default flex-direction is row, which makes the main axis horizontal (left to right in LTR languages). Understanding the axis model is essential because every alignment property in Flexbox is defined relative to either the main axis or the cross axis — if you don't know which axis you're operating on, justify-content and align-items become impossible to reason about.

2. flex-flow

Why: flex-flow is shorthand for flex-direction + flex-wrap. For example, flex-flow: row wrap sets both at once. This is useful in practice because you almost always need to think about direction and wrapping together — a column layout that wraps behaves very differently from a row layout that wraps.

3. align-self

Why: align-items on the container sets the cross-axis alignment for all items. But sometimes one item needs to differ — for example, a "new" badge that should sit at the top of a card while the rest of the content is centered. align-self overrides align-items for a single item. It accepts the same values: flex-start, flex-end, center, stretch, baseline.

4. nowrap

Why: The default flex-wrap: nowrap means all items are forced onto one line, even if they overflow the container. This is a deliberate design choice — Flexbox was created primarily for one-dimensional layouts. You must explicitly opt in to wrapping with flex-wrap: wrap. This is a common source of confusion: students expect items to wrap automatically like text, but they don't.

Question 2 — Multiple Choice (2 pts)

A developer writes the following CSS:

.toolbar {
  display: flex;
  flex-direction: row;
  justify-content: center;
}
.toolbar .logo {
  align-self: flex-start;
}

The toolbar contains a logo, a search bar, and a settings icon. The developer expects all items to be vertically aligned to the top of the toolbar. Instead, only the logo sits at the top while the other items stretch to fill the toolbar's height. Why does this happen?

  • A) justify-content: center overrides vertical alignment, pushing items to the middle
  • B) align-items defaults to stretch, so all items stretch vertically — the logo escapes this only because align-self: flex-start overrides the default for that one item
  • C) flex-direction: row disables vertical alignment entirely
  • D) The .logo rule has higher specificity, causing it to inherit a different box model
▸ Show Answer

B — align-items defaults to stretch, so all items stretch vertically. The logo overrides this with align-self: flex-start.

Why B is correct: When no align-items is set on a flex container, it defaults to stretch. This means every flex item expands to fill the container's cross-axis dimension (height, in a row layout). The .logo has align-self: flex-start, which overrides the container's align-items: stretch for that item only, pinning it to the top. The other items have no override, so they still stretch. To fix the toolbar, add align-items: flex-start (or center) to .toolbar.

Why the others are wrong:

  • A — justify-content operates on the main axis (horizontal in row). It has absolutely nothing to do with vertical alignment. This is the most common Flexbox misconception.
  • C — flex-direction: row does not disable vertical alignment. It simply defines the main axis as horizontal, which means the cross axis is vertical — and align-items / align-self control that cross axis.
  • D — Specificity determines which CSS rule wins when there are conflicts, but here there is no conflict. align-self on the item and align-items on the container are different properties targeting different elements.

Question 3 — Code Analysis: Explain what happens and why (6 pts)

Study the following CSS and HTML:

<div class="container">
  <div class="item a">A</div>
  <div class="item b">B</div>
  <div class="item c">C</div>
</div>
.container {
  display: flex;
  width: 600px;
}
.item { flex-basis: 100px; }
.a { flex-grow: 1; }
.b { flex-grow: 2; }
.c { flex-grow: 0; }

a) Explain what flex-grow does conceptually — not just "makes items bigger," but what specifically is being distributed and how. (2 pts)

b) Calculate the final width of each item. Show your reasoning step by step. (2 pts)

c) If item C had flex-grow: 0 but also flex-shrink: 0, how would its behaviour differ from the default when the container is narrower than the combined flex-basis of all items? Explain why. (2 pts)

▸ Show Answer

a) What flex-grow distributes:

flex-grow distributes the remaining free space in the container after all items have been sized at their flex-basis. It does not set proportional widths — a common misconception. The value is a ratio that determines how much of the leftover space each item claims. An item with flex-grow: 0 claims none of the leftover space and stays at its flex-basis size.

b) Calculation:

  • Container width: 600px
  • Total flex-basis: 100 + 100 + 100 = 300px
  • Remaining free space: 600 − 300 = 300px
  • Total grow factors: 1 + 2 + 0 = 3
  • Space per grow unit: 300 / 3 = 100px
  • Item A: 100 + (1 × 100) = 200px
  • Item B: 100 + (2 × 100) = 300px
  • Item C: 100 + (0 × 100) = 100px (stays at flex-basis)

Key insight: Item B is not "twice the width" of item A (300 vs 200). It received twice the extra space, but both started from the same 100px base. This distinction matters: if item A had flex-basis: 200px and item B had flex-basis: 50px, even with the same grow factors the final sizes would be very different.

c) flex-shrink: 0 behaviour:

By default, flex-shrink is 1, meaning all items will shrink proportionally when the container is too small. If item C has flex-shrink: 0, it refuses to shrink below its flex-basis (100px), even if the container overflows. Items A and B would absorb all the shrinking. This is useful in practice for elements that must maintain a fixed size — like an icon or a fixed-width sidebar — while other items adapt to the available space.

Question 4 — Select All That Apply (3 pts)

Which of the following statements about Flexbox are correct? For each statement, consider why it is true or false.

  • [ ] The float property has no effect on flex items
  • [ ] align-content works even when flex-wrap is set to nowrap
  • [ ] justify-content: space-between distributes space between items but not before the first or after the last
  • [ ] Setting flex: 1 on an item is shorthand for flex-grow: 1; flex-shrink: 1; flex-basis: 0
  • [ ] order: -1 on a flex item will visually move it before all items with the default order
  • [ ] Flexbox is a two-dimensional layout system that controls rows and columns simultaneously
▸ Show Answer

Correct statements:

  • [x] float has no effect on flex items — The Flexbox specification explicitly states that float and clear have no effect on flex items. This is because Flexbox establishes its own layout algorithm that completely replaces the normal block/inline flow. The item is taken out of float context. However, float still affects the element's display computation if the element is not a flex item.
  • [ ] align-content works with nowrap — FALSE. align-content distributes space between flex lines (rows of wrapped items). With nowrap, there is only one flex line, so there is nothing to distribute between. This is identical to how justify-content is meaningless if there's only one item — you need multiple things to distribute space between.
  • [x] justify-content: space-between distributes space between items — space-between places the first item flush to the start, the last item flush to the end, and distributes remaining space evenly between items. Compare with space-around (equal space on each side of every item) and space-evenly (equal gaps everywhere, including edges).
  • [x] flex: 1 equals flex-grow: 1; flex-shrink: 1; flex-basis: 0 — This is a critical detail. Note that flex-basis: 0 (not auto) means the item starts with zero size and grows to fill space. This is why flex: 1 on all items makes them equal width — they all start at 0 and share the space equally. If the shorthand were flex-basis: auto, items with different content widths would end up different sizes.
  • [x] order: -1 moves the item before default-order items — The default order value is 0. Items are laid out in ascending order. So order: -1 places the item before all items with order: 0. This is useful for accessibility: you can keep a logical DOM order for screen readers while changing visual order for sighted users.
  • [ ] Flexbox is two-dimensional — FALSE. Flexbox is a one-dimensional layout system. It handles either a row OR a column, never both simultaneously. CSS Grid is the two-dimensional system. Even when flex-wrap: wrap creates multiple rows, each row is laid out independently — items in row 2 don't align with columns in row 1.

Question 5 — Written Answer: Compare and Justify (4 pts)

Consider these two approaches to centering a <div> both horizontally and vertically inside its parent:

Approach A — Flexbox:

.parent {
  display: flex;
  justify-content: center;
  align-items: center;
  height: 100vh;
}

Approach B — Absolute positioning + transform:

.parent {
  position: relative;
  height: 100vh;
}
.child {
  position: absolute;
  top: 50%;
  left: 50%;
  transform: translate(-50%, -50%);
}

a) Explain how each approach achieves centering — what is the mechanism? (2 pts)

b) In what situation would you choose one over the other? Justify your choice with a concrete scenario. (2 pts)

▸ Show Answer

a) How each works:

Approach A (Flexbox): display: flex activates flex layout. justify-content: center centers items along the main axis (horizontal in the default row direction). align-items: center centers items along the cross axis (vertical). Together, these place the child at the exact center. The parent participates in the layout — it "knows" about its children and positions them.

Approach B (Absolute + transform): position: absolute removes the child from normal document flow. top: 50%; left: 50% places the child's top-left corner at the center of the parent. But this isn't true centering yet — the child is offset by its own width and height. transform: translate(-50%, -50%) shifts the child back by 50% of its own dimensions, moving the center of the child to the center of the parent.

b) When to choose each:

  • Choose Flexbox when you have multiple children to center or when the centered content should participate in the layout flow. For example, a hero section with a heading, subtitle, and button all centered vertically: Flexbox centers the group and lets you use gap between them. The children remain in flow and interact with each other predictably.
  • Choose absolute positioning when you need to overlay content on top of other content. For example, a "Loading..." spinner centered over an image: the spinner must be positioned independently of the image, not pushing it aside. Absolute positioning takes the spinner out of flow entirely, which is exactly what you want for an overlay.

General recommendation: Prefer Flexbox for most centering tasks. It is more maintainable, adapts to different content sizes, and doesn't require the parent-child position relationship. The absolute positioning approach is a legacy technique that still has its place for overlays and layered UIs.

SECTION B: CSS GRID

Question 6 — Fill in the blanks (4 pts)

Complete the following sentences about CSS Grid:

The property 1. ________ lets you define named layout regions so you can place elements visually, like mapping out a floor plan. The 2. ________ unit represents a fraction of the available free space in the grid container — it is not a fixed unit like px or %. To create a responsive grid that automatically adjusts the number of columns without media queries, you combine repeat() with 3. ________ and minmax() in grid-template-columns. The property 4. ________ controls the spacing between grid tracks (both rows and columns).

▸ Show Answer

1. grid-template-areas

Why: This property lets you "draw" your layout with named strings. Each string represents a row, and each word in the string names a cell. For example:

grid-template-areas:
  "header header"
  "sidebar main"
  "footer footer";

Then you assign elements to areas with grid-area: header. This is powerful because the layout is visually obvious in the code — you can see the structure just by reading the CSS. It also makes responsive redesign easy: change the area strings in a media query and the entire layout restructures without touching the HTML.

2. fr

Why: The fr unit divides remaining free space after fixed-size tracks are accounted for. So grid-template-columns: 250px 1fr means "the first column is exactly 250px, and the second column gets all the remaining space." This is fundamentally different from percentages: 1fr is calculated after gap and fixed-width columns are subtracted, while percentages are calculated from the container's total width (which can cause overflow when combined with gaps).

3. auto-fit (also acceptable: auto-fill)

Why: repeat(auto-fit, minmax(200px, 1fr)) tells the browser: "create as many columns as will fit, each at least 200px wide, and distribute remaining space equally." Without auto-fit, repeat() requires a fixed number like repeat(3, ...), which doesn't adapt to screen size. The difference between auto-fit and auto-fill is subtle: auto-fit collapses empty tracks so remaining items expand, while auto-fill keeps empty tracks, preserving minimum sizes.

4. gap (also acceptable: grid-gap)

Why: gap sets spacing between tracks (rows and columns). It replaces the older grid-gap and also works in Flexbox. The key advantage over margins is that gap only adds space between items, not around the outside edges. This eliminates the classic problem of needing to remove the margin from the last item in a row.

Question 7 — Code Analysis: Explain the layout and why (6 pts)

Study the following CSS from the teacher's course material:

.page {
  display: grid;
  grid-template-areas:
    "header header"
    "sidebar main"
    "footer footer";
  grid-template-columns: 250px 1fr;
}

a) Describe the layout this creates. How many rows and columns are there? What does each area occupy? (2 pts)

b) Explain what 1fr means in grid-template-columns: 250px 1fr. Why is 1fr better than calc(100% - 250px) for the second column? (2 pts)

c) A colleague suggests replacing the grid with Flexbox: display: flex; flex-wrap: wrap;. Explain why Flexbox would struggle with this specific layout and what problems would arise. (2 pts)

▸ Show Answer

a) Layout description:

This creates a 3-row, 2-column grid:

  • Row 1: The header spans both columns (the name "header" appears twice)
  • Row 2: The sidebar occupies the left column (250px wide), and main occupies the right column
  • Row 3: The footer spans both columns

Elements are placed by setting grid-area: header, grid-area: sidebar, etc. The visual structure in the CSS mirrors the actual page layout, making it self-documenting.

b) Why 1fr is better than calc(100% - 250px):

1fr means "one fraction of the remaining free space." The browser first allocates the fixed 250px to column 1, then gives everything that's left to column 2. Using calc(100% - 250px) seems equivalent, but it breaks when gap is added. If you set gap: 20px, the 1fr calculation automatically accounts for the 20px gap: the second column becomes container - 250px - 20px. With calc(100% - 250px), the gap is not subtracted, causing the total to exceed 100% and triggering overflow. The fr unit is gap-aware by design.

c) Why Flexbox struggles here:

Flexbox is one-dimensional — it manages items along a single axis. With flex-wrap: wrap, items flow into new rows, but each row is independent. The problems:

  • The sidebar and main content in row 2 need to align with specific column widths (250px + rest). In Flexbox, you'd have to manually set widths and hope they add up correctly. Grid inherently defines column tracks that all rows share.
  • Making the header and footer span "both columns" requires them to be width: 100% in Flexbox, which works, but the sidebar/main row requires careful flex-basis calculations that break when padding or gaps change.
  • The fundamental issue: this layout has both row AND column structure (header spans columns, sidebar has a fixed column width). That is a two-dimensional layout problem, which is exactly what Grid was designed for. Flexbox would require workarounds that Grid handles natively.

Question 8 — Multiple Choice (2 pts)

Consider this CSS:

.grid {
  display: grid;
  grid-template-columns: repeat(auto-fit, minmax(200px, 1fr));
  gap: 16px;
}

A student says: "This grid always creates exactly 3 columns because 1fr divides the space into thirds." Why is this incorrect?

  • A) auto-fit creates a fixed number of columns based on the number of child elements, not three
  • B) The number of columns depends on the container's width — auto-fit creates as many columns as will fit at their minimum size (200px), and 1fr distributes leftover space equally among them
  • C) minmax(200px, 1fr) means columns are always exactly 200px wide, so the number depends on the container divided by 200
  • D) repeat() always requires a number as its first argument; auto-fit defaults to 3
▸ Show Answer

B — The number of columns depends on the container's width. auto-fit creates as many columns as will fit at their minimum size (200px), and 1fr distributes leftover space equally among them.

Why B is correct: The expression repeat(auto-fit, minmax(200px, 1fr)) means: "Repeat columns as many times as will fit. Each column must be at least 200px wide (minimum) and can grow up to 1fr (maximum)." On an 800px-wide container, you'd get roughly 4 columns. On a 500px-wide container, roughly 2. This is the core mechanism for responsive grids without media queries. The 1fr maximum ensures leftover space is distributed, so columns grow beyond 200px rather than leaving a gap at the end.

Why the others are wrong:

  • A — auto-fit does not base the column count on the number of children. It bases it on how many columns of the minimum size can fit in the container. If you have 2 items but space for 5 columns, auto-fit creates tracks for 5 but collapses the empty ones (unlike auto-fill, which keeps them).
  • C — minmax(200px, 1fr) does not make columns exactly 200px. The 200px is the minimum; columns can grow up to 1fr. If only 2 columns fit, each will be much wider than 200px.
  • D — auto-fit is a valid keyword for repeat() and does not default to any number. It dynamically calculates the column count at render time.

Question 9 — Select All That Apply (3 pts)

Which of the following statements about CSS Grid are correct?

  • [ ] Grid items can span multiple rows and columns using grid-row and grid-column
  • [ ] grid-template-areas requires that every area name appears only once in the template
  • [ ] The fr unit divides the total container width, including space taken by fixed columns and gaps
  • [ ] Grid and Flexbox can be nested — a grid item can itself be a flex container
  • [ ] auto-fit and auto-fill behave identically in all cases
  • [ ] You can use a period (.) in grid-template-areas to define an empty cell
▸ Show Answer

Correct statements:

  • [x] Grid items can span multiple rows and columns — Using grid-column: 1 / 3 or grid-column: span 2, an item can occupy multiple tracks. This is fundamental to Grid — it's what makes Grid two-dimensional. You can also use named areas, where repeating the same name in grid-template-areas causes an element to span those cells.
  • [ ] Each area name appears only once — FALSE. The whole point of grid-template-areas is that an area name can appear in multiple cells to indicate spanning. In the teacher's example, "header header" makes the header span two columns. The name must form a rectangle — L-shapes or disjoint areas are not allowed.
  • [ ] fr divides the total container width — FALSE. fr divides the remaining free space after fixed tracks (like 250px) and gaps have been subtracted. This is the critical difference between fr and %. With grid-template-columns: 250px 1fr and gap: 20px, the 1fr column gets container - 250px - 20px, not container - 250px.
  • [x] Grid and Flexbox can be nested — This is the recommended approach from the teacher's material. Use Grid for the page-level macro layout, then use Flexbox inside individual components. For example, the main grid area might contain a flex container for card components: .card { display: flex; flex-direction: column; justify-content: space-between; }
  • [ ] auto-fit and auto-fill are identical — FALSE. They differ when there are fewer items than available tracks. auto-fill creates empty tracks that consume space. auto-fit collapses empty tracks to zero width, allowing items to stretch. With 2 items in an 800px container using minmax(200px, 1fr): auto-fill might create 4 tracks (items stay at ~200px), while auto-fit collapses the 2 empty tracks and lets the 2 items expand to ~400px each.
  • [x] A period (.) defines an empty cell — In grid-template-areas, a . represents a cell with no named area. For example: "header header" ". main" "footer footer" leaves the left cell of row 2 empty. This is useful for creating visual gaps in the layout without dummy elements.

SECTION C: GRID VS FLEXBOX — CHOOSING THE RIGHT TOOL

Question 10 — Written Answer: Compare and Justify (4 pts)

From the teacher's course material: "Grid = macro layout (2D control). Flexbox = micro layout (1D flow). They are complementary."

You are building a large e-commerce website. The design requires:

  • A complex page-level layout (header, sidebar, main content area, footer)
  • Flexible product card components inside the main content area
  • Each card has an image, title, price, and "Add to cart" button that should be pushed to the bottom

a) Explain how you would use CSS Grid and Flexbox together to implement this layout. Be specific about which tool handles which part, and write the key CSS declarations. (2 pts)

b) A colleague says "I'll just use Grid for everything, including the card internals." Explain one concrete problem they will encounter with the card layout that Flexbox handles better. (2 pts)

▸ Show Answer

a) Using Grid and Flexbox together:

CSS Grid for the page layout (two-dimensional, rows and columns):

.page {
  display: grid;
  grid-template-areas:
    "header header"
    "sidebar main"
    "footer footer";
  grid-template-columns: 250px 1fr;
}

Grid is the right choice here because the page layout is inherently two-dimensional: the header and footer span both columns, the sidebar has a fixed width, and the main content fills the remaining space. grid-template-areas makes the layout visually obvious and easy to restructure in media queries.

Flexbox for the card components (one-dimensional, vertical flow):

.card {
  display: flex;
  flex-direction: column;
  justify-content: space-between;
}

Flexbox is the right choice here because each card is a single-column layout: image on top, text in the middle, button at the bottom. flex-direction: column stacks the elements vertically, and justify-content: space-between pushes the button to the bottom regardless of how much text content there is.

For the card grid inside the main area, you might use Grid again:

.products {
  display: grid;
  grid-template-columns: repeat(auto-fit, minmax(250px, 1fr));
  gap: 20px;
}

b) Problem with Grid for card internals:

The card has variable content — some products have long titles, some have short ones. The "Add to cart" button must always sit at the bottom of the card, regardless of content height. With flex-direction: column and justify-content: space-between, Flexbox pushes the first child to the top and the last child to the bottom automatically, adapting to any content height.

With Grid, you'd need to define explicit row tracks: grid-template-rows: auto 1fr auto. This works, but it requires knowing the exact number of content sections. If a card variant adds a "Sale" badge or a rating, you must update the row template. Flexbox handles variable numbers of children naturally — space-between always pushes the last child to the bottom, no matter how many children exist. Flexbox is more flexible for component internals where content is dynamic.

Question 11 — Multiple Choice (2 pts)

You are building a navigation bar with a logo on the left, navigation links in the center, and a user avatar on the right. All items should be vertically centered. Which approach is most appropriate and why?

  • A) CSS Grid with grid-template-columns: 1fr 1fr 1fr — because you need three distinct sections
  • B) Flexbox with justify-content: space-between; align-items: center — because a navbar is a single row of items distributed along one axis
  • C) Absolute positioning for each element — because you need precise pixel control
  • D) float: left for the logo and links, float: right for the avatar
▸ Show Answer

B — Flexbox with justify-content: space-between; align-items: center.

Why B is correct: A navigation bar is a textbook case for Flexbox. It is a single row of items (one-dimensional), and you want to distribute them along the main axis while vertically centering them. justify-content: space-between pushes the logo to the left edge and the avatar to the right edge, with the links occupying the center. align-items: center vertically centers everything, handling items of different heights (a tall logo next to small text links).

Why the others are wrong:

  • A — Grid would work, but it's overkill. A navbar is not a two-dimensional layout. Using Grid for a single row of items adds unnecessary complexity. However, Grid is a valid choice if the navbar has more complex structure (e.g., items that need to align with grid columns in the content below).
  • C — Absolute positioning removes items from the flow, making the navbar unable to adapt to different content sizes. If the logo is wider than expected, it would overlap the links. Absolute positioning also makes responsive design much harder.
  • D — float was the pre-Flexbox solution for navbars. It requires clearfix hacks, doesn't support vertical centering, and makes it nearly impossible to center the middle group between the floated sides. Modern CSS has replaced float-based layouts entirely.

Question 12 — Code Analysis: Explain what happens and why (6 pts)

A student writes the following CSS to create a responsive card grid:

.cards {
  display: flex;
  flex-wrap: wrap;
  gap: 16px;
}
.card {
  flex: 1 1 300px;
}

Another student writes this alternative:

.cards {
  display: grid;
  grid-template-columns: repeat(auto-fit, minmax(300px, 1fr));
  gap: 16px;
}

a) Both approaches create responsive card grids. Explain how each one works — what happens when the viewport is wide? What happens when it narrows? (2 pts)

b) The Flexbox version has a well-known problem with the last row when there are, say, 5 cards in a 3-column layout. Describe this problem and explain why it occurs. (2 pts)

c) Which approach would you recommend for a product listing page and why? (2 pts)

▸ Show Answer

a) How each approach works:

Flexbox version: flex: 1 1 300px means each card starts at flex-basis: 300px, can grow (flex-grow: 1), and can shrink (flex-shrink: 1). With flex-wrap: wrap, cards flow into new rows when they can't fit. When the viewport is wide, multiple cards fit on one row and grow to fill the space. When the viewport narrows, fewer cards fit per row, and the cards on each row grow to fill that row.

Grid version: repeat(auto-fit, minmax(300px, 1fr)) creates as many columns as will fit at a minimum of 300px. When the viewport is wide, more columns are created. When it narrows, columns are removed and remaining columns grow. The key difference: Grid defines column tracks that are consistent across all rows.

b) The last-row problem in Flexbox:

With 5 cards in a 3-column layout: the first row has 3 cards, each taking ~33% of the width. The second row has only 2 cards. Because flex-grow: 1 distributes available space among the items in that row, the 2 remaining cards each grow to ~50% of the width. They become wider than the cards in the first row, breaking the visual grid alignment.

Why this happens: Flexbox is one-dimensional. Each row is an independent flex line. Items in row 2 have no knowledge of the column structure in row 1. They simply grow to fill their own row. Grid does not have this problem because column tracks span all rows — items always align to the same column widths.

c) Recommendation:

Use the Grid version for a product listing page. The last-row problem in Flexbox is a deal-breaker for card grids: cards changing width depending on which row they're in looks unprofessional. Grid ensures all cards maintain consistent column widths. Grid also handles the responsive column count automatically with auto-fit, without any additional calculation. The only reason to prefer Flexbox would be if you need cards to have varying widths based on their content, which is rarely desired in a product listing.

Question 13 — Written Answer: Explain reasoning (4 pts)

Consider the following two ways to make the card grid from Question 12 responsive on small screens:

Approach A — Media query:

.cards {
  display: grid;
  grid-template-columns: repeat(3, 1fr);
  gap: 16px;
}
@media (max-width: 768px) {
  .cards { grid-template-columns: repeat(2, 1fr); }
}
@media (max-width: 480px) {
  .cards { grid-template-columns: 1fr; }
}

Approach B — Intrinsic sizing:

.cards {
  display: grid;
  grid-template-columns: repeat(auto-fit, minmax(250px, 1fr));
  gap: 16px;
}

a) Explain how each approach handles responsiveness. What is the fundamental difference in strategy? (2 pts)

b) When would you choose Approach A over Approach B? Give a concrete design scenario where media queries are necessary. (2 pts)

▸ Show Answer

a) Fundamental difference:

Approach A (Media queries) uses extrinsic breakpoints: the developer decides at exactly which viewport widths the layout changes. At 769px you get 3 columns; at 768px it snaps to 2. The transitions are sharp and predefined. The developer must anticipate every screen size and write a rule for it.

Approach B (Intrinsic sizing) uses the content's own needs to determine the layout. The browser calculates how many 250px-minimum columns fit at any given width. There are no breakpoints — the layout adapts fluidly. A 900px container gets 3 columns; 800px still gets 3 (slightly narrower); at ~520px it drops to 2; at ~260px it drops to 1. The transitions are smooth and automatic.

The fundamental difference: Approach A puts the developer in control of when layout changes happen. Approach B puts the content and container in control. Approach B is often called "intrinsic design" — the layout responds to its own constraints, not arbitrary pixel breakpoints.

b) When Approach A is necessary:

Media queries are necessary when the layout change involves structural reorganization, not just column count. For example: on desktop, a dashboard shows a sidebar navigation + main content + detail panel (3-region layout). On tablet, the sidebar becomes a top navigation bar and the detail panel moves below the main content. On mobile, the navigation becomes a hamburger menu. These are not just "fewer columns" — they are completely different layouts that require different grid-template-areas at each breakpoint. auto-fit cannot restructure named areas; you need media queries to redefine them.

Question 14 — Select All That Apply (3 pts)

You are reviewing a colleague's CSS. Which of the following are valid criticisms of their code?

.layout {
  display: grid;
  grid-template-columns: 25% 75%;
  grid-gap: 20px;
}
  • [ ] Using % instead of fr will cause overflow when combined with grid-gap
  • [ ] grid-gap is deprecated and should be replaced with gap
  • [ ] The columns should add up to 100% but 25% + 75% = 100% is correct, so there is no problem
  • [ ] Using fr units instead of % would make the layout gap-aware and prevent overflow
  • [ ] The grid-gap property does not work in modern browsers
▸ Show Answer
  • [x] Using % instead of fr will cause overflow with grid-gap — This is correct and is a common bug. The percentages are calculated from the container's total width: 25% + 75% = 100%. But the 20px gap is added on top of the column widths, making the total 100% + 20px, which overflows the container. This is the same reason width: 50% on two floated elements breaks when you add a margin between them.
  • [x] grid-gap is deprecated and should be gap — grid-gap was the original property name, but the CSS Working Group renamed it to gap so it could also work in Flexbox. grid-gap still works in all browsers (for backward compatibility), but gap is the standard going forward. It's a valid criticism in a code review.
  • [ ] 25% + 75% = 100% means no problem — FALSE. This is exactly the trap. The percentages do add up to 100%, but the gap is additional space that pushes the total beyond 100%. If there were no gap, this would be fine. The fr unit avoids this because it divides only the remaining space after gaps.
  • [x] Using fr units would be gap-aware — Correct. grid-template-columns: 1fr 3fr would first subtract the 20px gap from the container width, then divide the remaining space in a 1:3 ratio. This is gap-aware by design and will never overflow.
  • [ ] grid-gap doesn't work in modern browsers — FALSE. It works perfectly in all modern browsers. It is deprecated (meaning you should prefer the newer name), but deprecated does not mean removed. Browser vendors maintain backward compatibility.

SECTION D: BEM & CSS ARCHITECTURE

Question 15 — Multiple Choice (2 pts)

Which of the following class names correctly follows BEM naming conventions?

  • A) .cardTitleActive
  • B) .card__title--active
  • C) .card-title-active
  • D) .Card__Title
▸ Show Answer

B — .card__title--active

Why B is correct: BEM stands for Block__Element--Modifier. The naming convention uses:

  • Block: The standalone component (card)
  • Element: A part of the block, separated by double underscores (__title)
  • Modifier: A variation or state, separated by double hyphens (--active)

BEM uses lowercase with hyphens for multi-word names (e.g., .search-form__submit-button--disabled).

Why the others are wrong:

  • A — .cardTitleActive uses camelCase, which is a JavaScript convention, not BEM. There are no separators to distinguish block, element, and modifier. In a large codebase, you can't tell if "Title" is an element of "card" or part of the block name.
  • C — .card-title-active uses single hyphens throughout. This is valid CSS and commonly used, but it is not BEM. BEM requires __ between block and element and -- between element and modifier. With single hyphens, you can't tell whether title is a separate word in the block name or an element.
  • D — .Card__Title uses the correct __ separator, but BEM convention requires lowercase. PascalCase is used in React component names, not CSS class names. Mixing conventions creates confusion in a codebase that uses both React and BEM.

Practical context: BEM solves the specificity and naming collision problem in large CSS codebases. By encoding the component hierarchy into the class name itself, BEM eliminates the need for nested selectors (like .card .title.active), keeping specificity flat and predictable.

Question 16 — Written Answer: Explain reasoning (4 pts)

A junior developer on your team writes the following CSS for a navigation component:

.nav { display: flex; gap: 8px; }
.nav ul { list-style: none; display: flex; gap: 16px; }
.nav ul li a { color: blue; text-decoration: none; }
.nav ul li a:hover { color: darkblue; }
.nav ul li a.active { color: red; font-weight: bold; }

a) Identify two problems with this CSS architecture (not visual bugs, but maintainability/scalability issues). (2 pts)

b) Rewrite the selectors using BEM naming conventions. Explain how BEM solves each problem you identified. (2 pts)

▸ Show Answer

a) Two architecture problems:

  1. Deep nesting / high specificity: The selector .nav ul li a has a specificity of 0-1-3 (one class, three element selectors). To override color: blue on a specific link, you'd need an even more specific selector. As the project grows, this leads to a "specificity war" where developers keep adding more selectors to win. Eventually someone resorts to !important, which makes things worse.
  2. Tight coupling to HTML structure: The selectors assume the exact DOM hierarchy: nav > ul > li > a. If you refactor the HTML (e.g., removing the ul and using <div> elements, or nesting the nav inside another component), all the CSS breaks. The styling is coupled to the markup structure rather than to the component's meaning.

b) BEM rewrite:

.nav { display: flex; gap: 8px; }
.nav__list { list-style: none; display: flex; gap: 16px; }
.nav__link { color: blue; text-decoration: none; }
.nav__link:hover { color: darkblue; }
.nav__link--active { color: red; font-weight: bold; }

How BEM solves each problem:

  • Flat specificity: Every selector is a single class (specificity 0-1-0). No nesting, no element selectors. Overriding any style requires just another single class. This makes the CSS predictable and easy to debug.
  • Structure independence: .nav__link targets any element with that class, regardless of its position in the DOM. You can refactor the HTML from <ul><li><a> to <div><a> without changing a single line of CSS. The class name describes the component relationship, not the DOM hierarchy.

Question 17 — Code Analysis: Explain the layout (4 pts)

Study the following complete layout that combines all the concepts from this exam:

<div class="page">
  <header class="page__header">...</header>
  <aside class="page__sidebar">...</aside>
  <main class="page__main">
    <div class="products">
      <article class="card">
        <img class="card__image" />
        <h3 class="card__title">Product</h3>
        <p class="card__price">$29</p>
        <button class="card__button">Add to cart</button>
      </article>
      <!-- more cards... -->
    </div>
  </main>
  <footer class="page__footer">...</footer>
</div>
.page {
  display: grid;
  grid-template-areas:
    "header header"
    "sidebar main"
    "footer footer";
  grid-template-columns: 250px 1fr;
  min-height: 100vh;
}
.page__header  { grid-area: header; }
.page__sidebar { grid-area: sidebar; }
.page__main    { grid-area: main; }
.page__footer  { grid-area: footer; }

.products {
  display: grid;
  grid-template-columns: repeat(auto-fit, minmax(250px, 1fr));
  gap: 20px;
}

.card {
  display: flex;
  flex-direction: column;
}
.card__button {
  margin-top: auto;
}

a) This layout uses Grid in two places and Flexbox in one place. For each usage, explain why that specific tool was chosen and what would go wrong if you swapped it for the other. (2 pts)

b) Explain what margin-top: auto does on the .card__button element in a flex column layout, and why this technique is preferred over using justify-content: space-between on the card in this case. (2 pts)

▸ Show Answer

a) Why each tool was chosen:

1. Grid for .page (page layout): The page has a two-dimensional structure — the header/footer span two columns while the sidebar/main split a row. Grid's grid-template-areas defines this perfectly. If you used Flexbox, you'd need the header, sidebar+main, and footer as three separate row groups, and making the sidebar a fixed 250px while main takes the rest would require nested flex containers. Grid handles the entire structure in one declaration.

2. Grid for .products (card grid): The product listing is a grid of cards that should have consistent column widths across all rows. repeat(auto-fit, minmax(250px, 1fr)) creates a responsive grid where every card is the same width. If you used Flexbox with flex-wrap: wrap, the last row (with fewer cards) would stretch those cards wider than the others — the "last row problem" from Question 12.

3. Flexbox for .card (card internals): Each card is a vertical stack of content (one-dimensional). The card's content varies in height (different title lengths), but the button must always be at the bottom. Flexbox with flex-direction: column handles this naturally. If you used Grid, you'd need to define explicit row tracks (grid-template-rows: auto auto auto auto), which is rigid and breaks if a card variant has more or fewer elements.

b) margin-top: auto in a flex column:

In a flex container, margin: auto absorbs all available free space in that direction. margin-top: auto on the button pushes it as far down as possible, consuming all the vertical space between the button and the content above it. This effectively "glues" the button to the bottom of the card.

Why this is preferred over justify-content: space-between:

justify-content: space-between would put equal space between all children: image, title, price, and button. This might push the title and price apart from each other, creating awkward spacing. With margin-top: auto on only the button, the image/title/price stay naturally grouped together (with their normal margins), and only the button is pushed to the bottom. It's more precise: you control which element gets the extra space, rather than distributing it among all children.

Question 18 — Select All That Apply (3 pts)

Which of the following statements about CSS custom properties (variables) are correct?

  • [ ] They can be updated dynamically with JavaScript at runtime
  • [ ] They are scoped and can be overridden within specific selectors
  • [ ] They replace the need for a CSS preprocessor like SASS entirely
  • [ ] They improve maintainability by centralising repeated values
  • [ ] They are natively supported in all CSS-in-JS libraries without any configuration
▸ Show Answer

Correct statements:

  • [x] They can be updated dynamically with JavaScript at runtime — CSS custom properties are live in the DOM. You can change them with element.style.setProperty('--color', 'red'), and all elements using that variable update immediately. This is fundamentally different from SASS variables, which are compiled away at build time and cannot change at runtime. This makes custom properties powerful for themes, dark mode toggles, and animations.
  • [x] They are scoped and can be overridden within specific selectors — Custom properties follow CSS's cascading and inheritance rules. If you define --color: blue on :root and --color: red on .sidebar, elements inside .sidebar will use red. This scoping is a key advantage over SASS variables, which are globally scoped (or block-scoped, but not DOM-scoped).
  • [ ] They replace SASS entirely — FALSE. CSS custom properties handle variables, but SASS also provides mixins, nesting, loops (@for, @each), functions, and modular file imports (@use). Custom properties cover one important feature of SASS, not all of them.
  • [x] They improve maintainability by centralising repeated values — Instead of writing #3498db in 40 places, you define --primary: #3498db once and reference it everywhere with var(--primary). Changing the primary color requires editing one line instead of 40. This is the same benefit SASS variables provide, but custom properties work at runtime in the browser.
  • [ ] They are natively supported in all CSS-in-JS libraries — FALSE. CSS custom properties are a browser feature, not a library feature. They work in any context where the browser parses CSS, including CSS-in-JS output. But saying they are "natively supported in CSS-in-JS libraries" implies those libraries have special handling for them, which is misleading. Each library has its own API for variables; some support CSS custom properties directly, others do not.
