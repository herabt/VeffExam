Question 1 — Multiple Choice

A developer builds a component where clicking a button should increment a counter displayed on screen. They store the count in a regular variable (let count = 0) instead of using useState. The variable increments correctly in the console, but the number on screen never changes. Why?

  • A) Regular variables cannot be changed inside event handlers
  • B) React does not know the variable changed — only useState setters schedule a re-render, so the component never updates its output
  • C) Regular variables are reset to their initial value on every render
  • D) The onClick handler runs on the server, not the client
▸ Show Answer

B — React components are functions that return JSX. The output only updates when React re-renders the component, and React only re-renders when state (via useState) changes. A regular variable changes in memory but React has no way to know it changed, so the screen stays the same.

Why C is partially true but not the root cause: It is true that a let variable inside a component resets on each render, but the real problem is that no re-render ever happens in the first place. If you used useState, React would re-render and the new state would persist.

Question 6 — Written Answer (4 pts)

A junior developer writes the following component:

function UserProfile({ userId }) {
    const [user, setUser] = useState(null);

    useEffect(() => {
        fetch(`/api/users/${userId}`)
            .then(res => res.json())
            .then(data => setUser(data));
    });

    return <div>{user?.name}</div>;
}

a) Identify the performance problem with this code and explain why it occurs. (2 pts)

b) How would you fix it? Show the corrected code and explain why your fix works. (2 pts)

▸ Show Answer

a) Performance problem: The useEffect has NO dependency array, which means it runs after EVERY render. Each time it runs, it calls setUser(data), which triggers a re-render, which runs the effect again → infinite loop of fetching.

b) Fix: Add [userId] as the dependency array:

useEffect(() => {
    fetch(`/api/users/${userId}`)
        .then(res => res.json())
        .then(data => setUser(data));
}, [userId]); // Only re-fetch when userId changes

This runs once on mount and only re-runs when userId actually changes.

Question 7 — Code Analysis (4 pts)

What is wrong with this component? Explain the issue and provide the correct implementation.

function Timer() {
    const [seconds, setSeconds] = useState(0);
    const intervalId = useState(null);

    useEffect(() => {
        intervalId = setInterval(() => {
            setSeconds(seconds + 1);
        }, 1000);

        return () => clearInterval(intervalId);
    }, []);

    return <div>Time: {seconds}s</div>;
}
▸ Show Answer

Multiple issues:

  1. intervalId uses useState(null) but is used as a variable — should be useRef(null) since we need a mutable value that persists without causing re-renders.
  2. setSeconds(seconds + 1) uses stale closure — seconds is captured from initial render (always 0). Should use functional update: setSeconds(prev => prev + 1).
  3. Direct assignment intervalId = ... won't work with useState's array return.

Correct implementation:

function Timer() {
    const [seconds, setSeconds] = useState(0);
    const intervalRef = useRef(null);

    useEffect(() => {
        intervalRef.current = setInterval(() => {
            setSeconds(prev => prev + 1); // functional update
        }, 1000);

        return () => clearInterval(intervalRef.current);
    }, []);

    return <div>Time: {seconds}s</div>;
}

SECTION B: STATE MANAGEMENT

Question 8 — Multiple Choice

A new developer on the team writes the following Redux reducer:

function cartReducer(state, action) {
    if (action.type === 'ADD_ITEM') {
        state.items.push(action.payload);
        return state;
    }
    return state;
}

The item appears to be added, but components using useSelector do not re-render. Why?

  • A) The reducer should use switch instead of if
  • B) The reducer mutates the existing state object and returns the same reference — Redux only re-renders when it detects a new state reference
  • C) push() does not work on arrays inside Redux state
  • D) The action type string must use the slice/action format
▸ Show Answer

B — Redux uses shallow reference comparison to detect state changes. state.items.push() mutates the existing array in place and return state returns the same object reference. Since the reference didn't change, Redux thinks nothing happened and skips re-rendering.

Fix: Return a new object with a new array: return { ...state, items: [...state.items, action.payload] }.

Note: Redux Toolkit's createSlice uses Immer, which lets you write state.items.push() safely — Immer intercepts the mutation and produces a new immutable state behind the scenes. But in a plain Redux reducer, direct mutation is a bug.

Question 10 — Written Answer (6 pts)

Describe the following Redux concepts. For each, explain what it does and how it fits within the Redux/Flux pattern:

a) Store (1.5 pts)
b) Actions (1.5 pts)
c) Reducers (1.5 pts)
d) Dispatch (1.5 pts)

▸ Show Answer

a) Store: The single source of truth for the entire application state. It holds the state tree and provides methods to access it (getState), update it (dispatch), and subscribe to changes. There is only ONE store in a Redux application.

b) Actions: Plain JavaScript objects that describe WHAT happened. They have a required type property (string) and an optional payload. Example: { type: 'cart/addItem', payload: { id: 1, name: 'Book' } }. Actions are the ONLY way to trigger state changes.

c) Reducers: Pure functions that take the current state and an action, and return a NEW state: (state, action) => newState. They determine HOW the state changes in response to actions. Must be pure — no side effects, no mutations, same input = same output.

d) Dispatch: The method used to send actions to the store. dispatch(action) triggers the reducer, which computes the new state. Components call dispatch to initiate state updates. It's the bridge between the component (View) and the reducer.

Question 20 — Written Answer (4 pts)

You are building a search input that filters a product list in real-time as the user types, AND a separate multi-field registration form that submits to a Server Action.

a) For the search input, would you use a controlled input (useState + onChange) or react-hook-form? Justify your choice. (2 pts)

b) For the registration form, would you use controlled inputs or react-hook-form? Justify your choice. (2 pts)

▸ Show Answer

a) Search input: controlled input (useState + onChange) — because the UI needs to react to every keystroke (filtering the product list in real time). You need the input value in state so that every change triggers a re-render and updates the filtered list. react-hook-form uses uncontrolled inputs that avoid re-renders, which is the opposite of what you want here.

b) Registration form: react-hook-form — because a multi-field form does not need to re-render the component on every keystroke. react-hook-form uses uncontrolled inputs, so the component only re-renders during validation events, not on every character typed. This is more performant. It also provides built-in validation rules (required, pattern, minLength), error display via formState.errors, and a resolver option for Zod schema validation.

Key principle: Use controlled inputs when the UI must react to input changes in real time. Use react-hook-form (uncontrolled) when you only need the values on submit and want to avoid unnecessary re-renders.
