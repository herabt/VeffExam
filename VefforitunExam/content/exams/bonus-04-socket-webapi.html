Question 2 — Multiple Choice

What is the difference between io.to(room).emit() and socket.to(room).emit() on the server?

  • A) They are identical — both send to all sockets in the room including the sender
  • B) io.to(room).emit() includes the sender; socket.to(room).emit() excludes the sender
  • C) socket.to(room).emit() can only be used inside a connection handler; io.to(room).emit() can be used anywhere
  • D) Both B and C are correct
▸ Show Answer

D — Both statements are true. io.to(room).emit() sends to all sockets in the room (including the sender if they are in the room). socket.to(room).emit() sends to everyone in the room except the socket it is called on. Question 3 — Code Analysis (6 pts)

Read the following Socket.io server code and answer the questions below:

io.on('connection', (socket) => {
  socket.on('joinRoom', (roomId, callback) => {
    socket.join(roomId);
    callback({ status: 'ok', room: roomId });
  });

  socket.on('chatMessage', ({ roomId, text }) => {
    socket.to(roomId).emit('newMessage', { text, from: socket.id });
  });

  socket.on('disconnect', (reason) => {
    console.log(`Socket ${socket.id} disconnected: ${reason}`);
  });
});

a) What is the purpose of the third argument (callback) in the joinRoom handler, and what Socket.io feature does this demonstrate? (2 pts)

b) When a client emits chatMessage, who receives the resulting newMessage event? (1 pt)

c) Name two common values the reason string might hold in the disconnect handler and what each means. (2 pts)

▸ Show Answer

a) callback demonstrates Socket.io acknowledgements. After joining the room, the server invokes the callback function with a response object — the client receives this response as confirmation. The client would emit like: socket.emit('joinRoom', roomId, (response) => console.log(response)). This gives a one-shot request/response pattern on top of the event model — no need to emit a separate "joinRoomSuccess" event.

b) Only the sockets in roomId except the sender receive newMessage. This uses socket.to(roomId) which excludes the emitting socket.

c) Common reasons:
"transport close" — the underlying transport (WebSocket/polling) was closed, typically because the client closed the browser tab or lost the connection.
"server namespace disconnect" — the server explicitly called socket.disconnect() to force the client off.
Other valid answers: "transport error" (network error), "ping timeout" (client stopped responding to heartbeats).

Question 5 — Written Answer (4 pts)

Explain how Socket.io reconnection works by default. What client-side options can you configure to control reconnection behavior? Give two examples with their effect.

▸ Show Answer

By default, Socket.io clients automatically attempt to reconnect when the connection is lost. The reconnection strategy uses exponential backoff — it waits an increasing amount of time between attempts to avoid overwhelming the server.

Configurable options (set on client at connection time):

  • reconnection: false — disables automatic reconnection entirely. Useful if you want to handle reconnection logic yourself.
  • reconnectionAttempts: 5 — limits how many times the client tries before giving up. After exhausting attempts the client emits reconnect_failed.
  • reconnectionDelay: 1000 — initial delay (ms) before the first reconnection attempt.
  • reconnectionDelayMax: 5000 — caps the maximum delay between attempts.

During reconnection the client emits events: reconnecting (with attempt number), reconnect (on success), and reconnect_failed (on giving up). Servers should emit any missed state on connection because Socket.io does not replay missed events.

Question 6 — Multiple Choice

A client emits an event with an acknowledgement callback. The server never calls the callback. What happens on the client side by default?

  • A) The client throws an unhandled Promise rejection
  • B) The callback is never called and the client has no built-in timeout for it
  • C) Socket.io automatically retries the emit after 5 seconds
  • D) The connection is closed and reconnection is triggered
▸ Show Answer

B — By default there is no built-in timeout for acknowledgements. If the server never calls the callback, it simply never fires. You must implement your own timeout logic (e.g. with setTimeout) or use the timeout() helper added in Socket.io v4: socket.timeout(5000).emit('event', data, callback) which will call the callback with a timeout error after 5 seconds if the server hasn't responded.

Question 7 — Code Writing (5 pts)

Write a minimal Socket.io server setup in Node.js that:

  1. Creates an HTTP server and attaches Socket.io to it with CORS allowed for http://localhost:3000
  2. Uses middleware to reject connections that do not provide a token in socket.handshake.auth
  3. On connection, logs the socket ID
▸ Show Answer 
const http = require('http');
const { Server } = require('socket.io');

const server = http.createServer();
const io = new Server(server, {
  cors: {
    origin: 'http://localhost:3000',
    methods: ['GET', 'POST']
  }
});

// Middleware — runs before every connection completes
io.use((socket, next) => {
  const token = socket.handshake.auth.token;
  if (!token) {
    return next(new Error('Authentication error: no token'));
  }
  // Optionally verify the token here...
  next();
});

io.on('connection', (socket) => {
  console.log('Socket connected:', socket.id);
});

server.listen(3001, () => console.log('Listening on port 3001'));

Key points: new Server(httpServer, options) attaches Socket.io. CORS is configured in the constructor options. io.use() registers middleware — calling next(new Error(...)) rejects the connection and the client receives a connect_error event.

SECTION B: WEBSOCKETS VS HTTP VS SSE VS LONG POLLING

Question 8 — Multiple Choice

Which of the following best describes Server-Sent Events (SSE)?

  • B) A unidirectional server-to-client push mechanism over a persistent HTTP connection
  • C) A polling strategy where the client sends a request every few seconds

Question 10 — Written Answer (4 pts)

A product manager asks: "We need live score updates for a sports app. Users never need to send data back except for subscribing to a team." Compare WebSockets and SSE for this use case. Which would you recommend and why?

▸ Show Answer

Recommendation: SSE — because the data flow is unidirectional (server pushes scores, client only "subscribes" once via a URL parameter or query string). SSE fits perfectly here.

SSE advantages for this use case:

  • Simpler to implement — just a standard HTTP endpoint returning text/event-stream
  • Works through HTTP/2 multiplexing with no extra library
  • Built-in browser reconnection with Last-Event-ID resume
  • Easier load-balancer/proxy compatibility (plain HTTP)

When WebSockets would be better: if users could also chat, place bets, or interact in real time — i.e. when bidirectional communication is required.

Question 14 — Written Answer (4 pts)

Explain the difference between 401 Unauthorized and 403 Forbidden. Give a concrete scenario for each.

▸ Show Answer

401 Unauthorized — despite its name, this means unauthenticated. The server does not know who is making the request (no credentials provided, or the provided token/session is invalid/expired). The HTTP spec requires the response to include a WWW-Authenticate header explaining how to authenticate. Scenario: A user tries to access GET /api/profile without sending an Authorization header at all — the server returns 401 saying "you need to log in first."

403 Forbidden — the server knows who the user is (they are authenticated) but the user does not have permission to access the resource. Scenario: A logged-in regular user tries to access DELETE /api/admin/users/5 — the server recognizes them but they are not an admin, so it returns 403 saying "you are not allowed to do this."

Question 17 — Multiple Choice

A frontend at https://app.example.com sends a PUT request with a custom header X-Request-ID to https://api.example.com. The server has CORS configured correctly. What will the browser do before the actual PUT?

  • A) Nothing — PUT is a simple method and no preflight is needed
  • B) Send an OPTIONS preflight to https://api.example.com to check if PUT and X-Request-ID are allowed
  • C) Block the request entirely because cross-origin PUT is always forbidden
  • D) Send the PUT directly but strip the custom header for security
▸ Show Answer

B — A preflight is triggered because: (1) PUT is not a CORS-simple method (only GET, HEAD, POST are simple), and (2) X-Request-ID is not a CORS-safelisted request header. The browser sends OPTIONS with Access-Control-Request-Method: PUT and Access-Control-Request-Headers: x-request-id. Only if the server responds with the appropriate Access-Control-Allow-* headers will the actual PUT proceed.

Question 18 — Written Answer (4 pts)

A developer sets Access-Control-Allow-Origin: * on their API. A frontend tries to send a request with credentials: 'include' (cookies). The request fails. Explain why, and describe exactly what changes must be made on both the server and client to fix it.

▸ Show Answer

Why it fails: The CORS spec explicitly prohibits the wildcard (*) for Access-Control-Allow-Origin when Access-Control-Allow-Credentials: true is also needed. Browsers enforce this — using * with credentialed requests is treated as an error, and the browser blocks the response.

Server-side fixes:

  1. Change Access-Control-Allow-Origin from * to the exact origin: Access-Control-Allow-Origin: https://app.example.com
  2. Add the header: Access-Control-Allow-Credentials: true
  3. Optionally add Vary: Origin so caches do not serve the wrong origin's response

Client-side fix: the fetch must already include credentials: 'include' (or withCredentials: true for XHR) — if this is already present, no additional client change is needed.

Question 22 — Written Answer (5 pts)

Compare session-based authentication and JWT-based (token) authentication. Discuss: where state lives, scalability implications, and one security consideration for each.

▸ Show Answer

Session-based:

  • State location: Server stores session data (usually in memory or a database like Redis). The client holds only a session ID (typically in a cookie).
  • Scalability: Stateful — if the app runs on multiple servers, session store must be shared (e.g. Redis), or sticky sessions must be used. This adds infrastructure complexity.
  • Security consideration: Session IDs in cookies should use HttpOnly; Secure; SameSite=Lax. Session fixation and hijacking are risks if IDs are not rotated on login.

JWT-based:

  • State location: Stateless — all user info is encoded in the token itself. The server does not store anything per-user.
  • Scalability: Easier horizontal scaling — any server can validate the token using the shared secret/public key, no shared session store needed.
  • Security consideration: JWTs cannot be invalidated before expiry without a server-side token revocation list (effectively making them stateful). Storing JWTs in localStorage exposes them to XSS. Prefer HttpOnly cookies for JWT storage.

Question 24 — Written Answer (4 pts)

Explain how a Cross-Site Scripting (XSS) attack works, what data it can steal, and describe two complementary defenses.

▸ Show Answer

How XSS works: An attacker injects malicious JavaScript into a web page viewed by other users — either via stored input (stored XSS: comment saved to DB that renders unescaped), reflected input (reflected XSS: URL parameter echoed into HTML), or DOM manipulation (DOM XSS: client-side code writes unsanitized data to the DOM). The injected script runs in the victim's browser under the site's origin.

What it can steal: Anything accessible in the DOM or JavaScript context — localStorage, sessionStorage, non-HttpOnly cookies, form inputs, auth tokens, CSRF tokens, keystrokes.

Two defenses:

  1. Output encoding / escaping — HTML-encode user-supplied data before inserting it into HTML. Never use innerHTML with untrusted data; use textContent instead. Use templating libraries that auto-escape.
  2. Content Security Policy (CSP) — an HTTP response header (Content-Security-Policy) that tells the browser which sources of scripts are trusted. A strict CSP (e.g. script-src 'self') blocks inline scripts and external scripts from unauthorised domains, limiting the damage even if injection occurs.

Question 25 — Written Answer (4 pts)

Explain how a Cross-Site Request Forgery (CSRF) attack works. What makes it possible, and describe two mitigations.

▸ Show Answer

How CSRF works: The attacker crafts a malicious page that silently makes a state-changing HTTP request (e.g. POST /transfer-money) to a site where the victim is already logged in. Because the browser automatically attaches cookies for that origin, the server sees a request with valid session credentials and processes it — even though the user never intended to make the request.

What makes it possible: Cookies are sent automatically with every same-origin request from the browser, regardless of where the page initiating the request came from.

Two mitigations:

  1. SameSite cookie attribute — Setting SameSite=Lax or SameSite=Strict on session cookies prevents the browser from sending them on cross-origin requests, stopping most CSRF vectors.
  2. CSRF tokens (synchronizer token pattern) — the server embeds a unique, unpredictable token in forms/responses. The client must echo it back in subsequent requests (e.g. in a header or request body). An attacker on a different origin cannot read the token (blocked by CORS/SOP) and thus cannot forge a valid request.

Question 28 — Written Answer (4 pts)

A developer builds a modal dialog with a <div> overlay. List four specific accessibility requirements a modal must fulfill for it to be usable by keyboard and screen reader users.

▸ Show Answer
  1. Focus trap — while the modal is open, Tab and Shift+Tab should cycle only through focusable elements inside the modal. Focus must not escape to the background page.
  2. Initial focus — when the modal opens, focus must be moved into it (typically to the first focusable element or the modal container with tabindex="-1").
  3. Return focus — when the modal closes, focus must return to the element that triggered it (e.g. the button that opened it).
  4. ARIA role and labeling — the modal container needs role="dialog" (or role="alertdialog" for critical messages), aria-modal="true", and an accessible name via aria-labelledby pointing to the modal's heading.
  5. Bonus: Escape key should close the modal.
  6. Bonus: Background content should be inert (inert attribute or aria-hidden="true") so screen readers cannot accidentally navigate to it.

Question 36 — Multiple Choice

Question 37 — Multiple Choice

Which of the following correctly uses the FormData API to send a file upload via fetch?

  • A) fetch('/upload', { method: 'POST', headers: { 'Content-Type': 'multipart/form-data' }, body: JSON.stringify(formData) })
  • B) fetch('/upload', { method: 'POST', body: formData })
  • C) fetch('/upload', { method: 'POST', headers: { 'Content-Type': 'application/json' }, body: formData })
  • D) fetch('/upload', { method: 'GET', body: formData })
▸ Show Answer

B — When you pass a FormData object as the body, the browser automatically sets the correct Content-Type: multipart/form-data; boundary=... header including the boundary string. You must NOT set the Content-Type manually because the boundary would be missing and the server could not parse the body. Option A JSON-stringifies the FormData (produces "{}"), option C sets the wrong content type, option D uses GET (GET requests cannot have a body).

Question 38 — Written Answer (3 pts)

Explain why you should NOT manually set Content-Type: multipart/form-data when sending a FormData object with fetch.

▸ Show Answer

The multipart/form-data content type requires a boundary parameter that separates the different parts of the request body (e.g. Content-Type: multipart/form-data; boundary=----WebKitFormBoundaryXYZ). This boundary string is generated by the browser and embedded throughout the request body.

If you manually set Content-Type: multipart/form-data without the boundary, the header is malformed — the server cannot determine where each part starts and ends, causing parsing failures. By omitting the Content-Type header entirely when using FormData, you let the browser set the complete header including the correct boundary automatically.

Question 41 — Written Answer (4 pts)

Explain the Mediator pattern. How does it differ from the Observer pattern? Give a concrete JavaScript/web example of each.

▸ Show Answer

Observer: Objects (observers) subscribe to a subject/emitter and react when the subject emits an event. The subject broadcasts to all subscribers — it does not know what each subscriber will do. Coupling is between each observer and the subject. Example: React's custom event bus (mitt) — a component publishes 'cartUpdated' and multiple other components subscribe independently.

Mediator: A central coordinator (mediator) handles all communication between components. Components do not communicate directly with each other — they send messages to the mediator and the mediator decides who to notify. This reduces the number of direct dependencies between components. Example: A Redux store or an RTK store.dispatch — UI components never call each other directly; they all dispatch to the store (mediator) which updates state and re-renders the correct components. A chat room server is also a classic mediator: users don't send messages to each other directly, they send to the server which routes messages.

Key difference: Observer = many objects watching one subject; Mediator = one coordinator managing communication between many objects.

Question 43 — Code Analysis (4 pts)

Consider the following code that uses the Decorator pattern:

class Logger {
  log(message) {
    console.log(message);
  }
}

class TimestampDecorator {
  constructor(logger) {
    this.logger = logger;
  }
  log(message) {
    this.logger.log(`[${new Date().toISOString()}] ${message}`);
  }
}

class LevelDecorator {
  constructor(logger, level) {
    this.logger = logger;
    this.level = level;
  }
  log(message) {
    this.logger.log(`[${this.level}] ${message}`);
  }
}

const logger = new LevelDecorator(new TimestampDecorator(new Logger()), 'INFO');
logger.log('Server started');

a) What will be printed to the console? (2 pts)

b) Explain the advantage of this approach compared to adding timestamp and level functionality directly to the Logger class. (2 pts)

▸ Show Answer

a) [INFO] [2024-xx-xxTxx:xx:xx.xxxZ] Server started (with a real ISO timestamp). The call chain is: LevelDecorator.log prepends [INFO] and calls TimestampDecorator.log, which prepends a timestamp and calls Logger.log, which calls console.log.

b) The Decorator pattern allows composing behaviors dynamically without modifying the base class. Advantages:

  • Open/Closed Principle — the base Logger class is closed for modification but open for extension via decorators.
  • Mix and match — you can use TimestampDecorator alone, LevelDecorator alone, both together, or in any order, without creating a combinatorial explosion of subclasses.
  • Adding the behavior directly to Logger would bloat the class and violate Single Responsibility; it would also be impossible to use the logger without those features.

Question 44 — Written Answer (4 pts)

Explain the Flyweight pattern. Describe a concrete scenario in a web application where applying it would provide significant memory savings.

▸ Show Answer

Flyweight pattern: Instead of creating a new object for each similar entity, the pattern shares a single instance of the common (intrinsic) state across many objects. The unique (extrinsic) state is stored externally and passed in when needed.

Concrete web scenario: A mapping application renders 100,000 map markers, each representing a point of interest. Each marker has: a type (restaurant, hotel, park — intrinsic/shared) and its GPS coordinates + name (extrinsic/unique). Without Flyweight, 100,000 objects would each contain a copy of the icon, style, and behavior for their type. With Flyweight, only 3 shared "type" objects exist (Restaurant, Hotel, Park), and each marker holds only its coordinates/name plus a reference to the shared type. Memory usage drops dramatically.
