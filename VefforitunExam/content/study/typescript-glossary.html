TypeScript Quick-Reference Glossary

A dedicated reference covering every TypeScript concept tested in Web Programming II. Every entry includes a definition and a short code snippet.

1. Basic Types

TypeScript adds static types on top of JavaScript. All type annotations are stripped at compile time — they exist only for the developer and the compiler.

Type Description Example
stringText valueslet name: string = "Arnar";
numberAll numbers (int & float)let age: number = 30;
booleantrue / falselet active: boolean = true;
number[] / Array<number>Array of numbers (two equivalent syntaxes)let scores: number[] = [90, 85];
[string, number]Tuple — fixed length, fixed types per positionlet entry: [string, number] = ["age", 30];
enumNamed set of constantsenum Dir { Up, Down }
anyOpts out of type checking entirely — avoid!let x: any = "hello"; x = 42;
unknownLike any but safe — must narrow before uselet v: unknown = getData();
neverA value that never occurs (throwing fn, infinite loop)function fail(): never { throw new Error(); }
voidFunction returns nothing (returns undefined)function log(m: string): void { console.log(m); }
nullExplicitly null (with strictNullChecks)let n: null = null;
undefinedExplicitly undefinedlet u: undefined = undefined;
objectAny non-primitive — rarely used directlylet o: object = { x: 1 };

Example

let name: string = "Arnar";
let age: number = 30;
let active: boolean = true;
let nothing: null = null;
let anything: any = "works"; anything = 42; // no error
let safe: unknown = getData();
// safe.toUpperCase(); // ERROR — must narrow first
if (typeof safe === "string") {
  console.log(safe.toUpperCase()); // OK — narrowed to string
}

function crash(): never {
  throw new Error("boom");
}
function greet(): void {
  console.log("hi");
}
greet(); // "hi"

2. Type Annotations vs Type Inference

Type annotation — you explicitly write the type with a colon (: Type).

Type inference — TypeScript reads the value and figures out the type automatically. You do not always need to write annotations.

Example

// Type annotation — explicit
let city: string = "Reykjavik";
let count: number = 5;

// Type inference — no annotation needed
let city2 = "Reykjavik"; // inferred: string
let count2 = 5;           // inferred: number
let flag = true;          // inferred: boolean

city2 = 42; // ERROR: Type 'number' is not assignable to type 'string'

// Inference also works for return types:
function add(a: number, b: number) {
  return a + b; // return type inferred as number
}
console.log(add(3, 4)); // 7

When TypeScript cannot infer the type and no annotation is given, it falls back to any. With noImplicitAny: true in tsconfig, this becomes an error.

3. interface vs type

Both describe the shape of an object. The differences matter at the edges. In the course, interface is the default choice for object/class contracts; type is used for unions, literal strings, and derived types.

Feature interface type
Object shapesYesYes
Union types (A | B)NoYes
Primitive aliasesNoYes (type Name = string)
Declaration merging (reopen)Yes — declare twice, they mergeNo — second declaration = error
Extension / inheritanceextends& intersection
implements in classesYesYes
When to preferObject / class contracts, library APIsUnions, computed types, primitives

Example

// interface — preferred for object shapes and class contracts
interface User {
  id: number;
  name: string;
}
interface User { email: string; } // declaration merging — valid!
// User is now { id, name, email }

// Extending an interface (extends keyword):
interface Admin extends User { role: string; }

// Both work with implements:
class RegularUser implements User {
  id = 1;
  name = "Arnar";
  email = "a@a.is";
}

// type — preferred for unions, literal strings, and derived shapes
type Result = "success" | "error" | "loading"; // union — interface cannot do this
type ID = string | number;
type Point = { x: number; y: number };

// Extending with type uses intersection (&):
type AdminT = User & { role: string }; // same shape as Admin above

4. Union and Intersection Types

Union (A | B) — the value can be one of the listed types. Use |.

Intersection (A & B) — the value must satisfy all of the listed types simultaneously. Use &.

Example

// Union — ID can be string OR number
type ID = string | number;
function printId(id: ID) {
  if (typeof id === "string") {
    console.log(id.toUpperCase()); // string branch
  } else {
    console.log(id.toFixed(2));    // number branch
  }
}
printId("abc");
printId(42);

// Nullable union (common pattern)
function greet(name: string | null) {
  if (name === null) return "Hello, stranger!";
  return `Hello, ${name}!`;
}
console.log(greet("Arnar")); // "Hello, Arnar!"
console.log(greet(null));    // "Hello, stranger!"

// Intersection — merged shape
type Timestamped = { createdAt: Date };
type WithId = { id: number };
type TimestampedWithId = Timestamped & WithId; // { createdAt: Date; id: number }

5. Literal Types and Narrowing

Literal type — the type is an exact value, not just a category (e.g., "left" not just string).

Narrowing — TypeScript refines the type inside a conditional block based on a type guard.

Example

// Literal type — only these exact strings are valid
type Direction = "left" | "right" | "up" | "down";
function move(dir: Direction) {
  console.log(dir);
}
move("left");      // "left"
// move("diagonal"); // ERROR: not in the literal union

// Narrowing with typeof
function format(val: string | number): string {
  if (typeof val === "string") return val.toUpperCase(); // string here
  return val.toFixed(2);                                 // number here
}
console.log(format("hello")); // "HELLO"
console.log(format(3.14));    // "3.14"

// Narrowing with equality check
function handleStatus(status: "open" | "closed") {
  if (status === "open") {
    console.log("still open");
  } else {
    console.log("closed"); // TypeScript knows it must be "closed"
  }
}
handleStatus("open");   // "still open"
handleStatus("closed"); // "closed"

6. Optional (?) and readonly Properties

Optional property (prop?: Type) — the property may be present or absent (its type becomes Type | undefined).

Readonly property (readonly prop: Type) — the property can be set on creation but cannot be reassigned afterwards.

Example

interface User {
  id: number;
  name: string;
  email?: string;          // optional — string | undefined
  readonly createdAt: Date; // can't change after creation
}

const u: User = { id: 1, name: "Arnar", createdAt: new Date() };
u.name = "New Name";       // OK — writable
u.createdAt = new Date();  // ERROR: Cannot assign to 'createdAt' (readonly)

function greet(user: User) {
  // Must handle possibly-undefined optional
  if (user.email) {
    console.log(user.email.toUpperCase());
  }
  // Or use optional chaining:
  console.log(user.email?.toUpperCase());
}

7. Function Types, Overloads, Rest & Default Parameters

Function type annotations

// Parameter types + return type
function add(a: number, b: number): number {
  return a + b;
}

// Arrow function type
const multiply: (x: number, y: number) => number = (x, y) => x * y;

// Function type alias
type Transformer = (input: string) => string;
const upper: Transformer = (s) => s.toUpperCase();

// void return (no meaningful return)
function log(msg: string): void {
  console.log(msg);
}

Default parameters

function greet(name: string, greeting: string = "Hello"): string {
  return `${greeting}, ${name}!`;
}
console.log(greet("Arnar"));          // "Hello, Arnar!"
console.log(greet("Arnar", "Hi"));   // "Hi, Arnar!"

Rest parameters

function sum(...nums: number[]): number {
  return nums.reduce((a, b) => a + b, 0);
}
console.log(sum(1, 2, 3, 4)); // 10

Function overloads

Declare multiple signatures for different argument patterns. The implementation signature (last one) handles all cases but is not directly callable.

// Overload signatures
function format(val: string): string;
function format(val: number): string;
// Implementation signature (not directly callable)
function format(val: string | number): string {
  if (typeof val === "string") return val.toUpperCase();
  return val.toFixed(2);
}
console.log(format("hello")); // "HELLO"
console.log(format(3.14));    // "3.14"

8. Generics

Generics make components reusable across many types while preserving type safety. The type parameter T is supplied by the caller.

Generic function

function identity<T>(value: T): T {
  return value;
}
console.log(identity<string>("hello")); // "hello"
console.log(identity<number>(42));      // 42
console.log(identity("hello"));          // T inferred as string

Generic interface / type

interface ApiResponse<T> {
  data: T;
  status: number;
  message: string;
}

const userRes: ApiResponse<User> = {
  data: { id: 1, name: "Arnar" },
  status: 200,
  message: "OK",
};
console.log(userRes.data.name); // "Arnar"

Generic class

class Box<T> {
  private value: T;
  constructor(val: T) { this.value = val; }
  get(): T { return this.value; }
}
const strBox = new Box("hello");  // Box<string>
const numBox = new Box(42);       // Box<number>
console.log(strBox.get()); // "hello"
console.log(numBox.get()); // 42

Constraints with extends

// T must have a .length property
function logLength<T extends { length: number }>(val: T): void {
  console.log(val.length);
}
logLength("hello");   // 5
logLength([1, 2, 3]); // 3
// logLength(42);     // ERROR: number has no .length

keyof constraint

// K must be a key of T
function getProperty<T, K extends keyof T>(obj: T, key: K): T[K] {
  return obj[key];
}
const user = { id: 1, name: "Arnar" };
console.log(getProperty(user, "name")); // "Arnar"
console.log(getProperty(user, "id"));   // 1
// getProperty(user, "age");            // ERROR: not a key of user

infer (advanced)

// infer captures a type variable inside a conditional type
type UnpackPromise<T> = T extends Promise<infer U> ? U : T;
// UnpackPromise<Promise<string>> resolves to string
// UnpackPromise<number>          resolves to number (not a Promise, returns as-is)

9. Utility Types

TypeScript ships built-in generic types that transform existing types. The core ones (Partial, Pick, Omit, Record, Required, Readonly) are exam favourites. Ones marked bonus are less commonly covered in class.

Base type used in examples below

interface User {
  id: number;
  name: string;
  email: string;
  age: number;
}
Utility What it does Before → After
Partial<T>All properties become optionalUser → { id?: number; name?: string; ... }
Required<T>All properties become required (reverses Partial)Partial<User> → User
Readonly<T>All properties become readonlyUser → all fields are readonly
Pick<T, K>Keep only the listed keysPick<User, "id" | "name"> → { id, name }
Omit<T, K>Remove the listed keysOmit<User, "email"> → { id, name, age }
Record<K, V>Object with keys of type K and values of type VRecord<string, User> → { [key: string]: User }
Exclude<T, U>Remove types from a unionExclude<"a"|"b"|"c", "a"> → "b" | "c"
Extract<T, U>Keep only types that appear in bothExtract<"a"|"b"|"c", "a"|"c"> → "a" | "c"
NonNullable<T>Remove null and undefined from a unionNonNullable<string|null|undefined> → string
ReturnType<T> (bonus)The return type of a function typeReturnType<typeof add> → number
Parameters<T> (bonus)Tuple of parameter types of a functionParameters<typeof add> → [number, number]
Awaited<T> (bonus)Unwrap a Promise type recursivelyAwaited<Promise<string>> → string

Example

// Partial — useful for update/patch functions
function updateUser(id: number, updates: Partial<User>): void {
  console.log(`Updating user ${id}:`, updates);
}
updateUser(1, { name: "New Name" }); // OK — no need to pass all fields

// Pick — expose a limited view
type UserPreview = Pick<User, "id" | "name">;
// { id: number; name: string }
const preview: UserPreview = { id: 1, name: "Arnar" };
console.log(preview); // { id: 1, name: "Arnar" }

// Omit — hide sensitive fields
type PublicUser = Omit<User, "email">;
// { id: number; name: string; age: number }

// Record — map of users by string key (used heavily in the course)
const cache: Record<string, User> = {};
cache["u1"] = { id: 1, name: "Arnar", email: "a@a.is", age: 30 };
console.log(cache["u1"].name); // "Arnar"

// Required — reverses Partial (all fields mandatory again)
type StrictUser = Required<Partial<User>>; // same as User

// Bonus: ReturnType / Parameters
function add(a: number, b: number): number { return a + b; }
type AddReturn = ReturnType<typeof add>;     // number
type AddParams = Parameters<typeof add>;     // [number, number]

// Bonus: NonNullable
type MaybeName = string | null | undefined;
type DefiniteName = NonNullable<MaybeName>;  // string

// Bonus: Awaited
type FetchResult = Awaited<Promise<User[]>>; // User[]

10. Type Guards, Discriminated Unions

typeof guard

function process(val: string | number) {
  if (typeof val === "string") {
    return val.toUpperCase(); // string here
  }
  return val.toFixed(2);     // number here
}
console.log(process("hello")); // "HELLO"
console.log(process(3.14));    // "3.14"

instanceof guard

class Cat { meow() { return "meow"; } }
class Dog { bark() { return "woof"; } }

function speak(animal: Cat | Dog) {
  if (animal instanceof Cat) {
    return animal.meow(); // Cat here
  }
  return animal.bark();   // Dog here
}
console.log(speak(new Cat())); // "meow"
console.log(speak(new Dog())); // "woof"

in guard

interface Fish { swim(): void; }
interface Bird { fly(): void; }

function move(creature: Fish | Bird) {
  if ("swim" in creature) {
    creature.swim(); // Fish here
  } else {
    creature.fly();  // Bird here
  }
}

User-defined type guard (is)

function isString(val: unknown): val is string {
  return typeof val === "string";
}

function process(val: unknown) {
  if (isString(val)) {
    console.log(val.toUpperCase()); // string here — TypeScript trusts the guard
  }
}
process("hello"); // "HELLO"
process(42);      // (no output — not a string)

Discriminated unions

A pattern where each member of a union has a literal discriminant property (e.g., kind or type). TypeScript narrows automatically by checking that property.

interface Square { kind: "square"; side: number; }
interface Circle { kind: "circle"; radius: number; }
type Shape = Square | Circle;

function area(shape: Shape): number {
  switch (shape.kind) {
    case "square": return shape.side ** 2;   // Square here
    case "circle": return Math.PI * shape.radius ** 2; // Circle here
  }
}
console.log(area({ kind: "square", side: 4 }));         // 16
console.log(area({ kind: "circle", radius: 5 }));       // ~78.54

11. as Type Assertion, as const, Non-null Assertion !

as — type assertion

Tells TypeScript "trust me, I know the type." Does not change the runtime value.

// DOM query returns HTMLElement | null — assert to specific type
const input = document.getElementById("name") as HTMLInputElement;
input.value = "Arnar"; // OK — TypeScript knows it's HTMLInputElement

// Alternative angle-bracket syntax (not allowed in .tsx files):
const input2 = <HTMLInputElement>document.getElementById("name");

// Double assertion — use sparingly (escape hatch)
const num = ("42" as unknown) as number;

as const — const assertion

Makes an object or array deeply readonly and narrows all values to their literal types.

const colors = ["red", "green", "blue"] as const;
// type: readonly ["red", "green", "blue"]
// without as const: type would be string[]

const config = { host: "localhost", port: 3000 } as const;
// type: { readonly host: "localhost"; readonly port: 3000 }
config.port = 4000; // ERROR: readonly

Non-null assertion !

Tells TypeScript the value is definitely not null or undefined. Use when you are certain — TypeScript cannot verify this at runtime.

const el = document.getElementById("root")!; // asserts non-null
el.innerHTML = "<p>Hello</p>"; // OK — no null check needed

// Without !:
const el2 = document.getElementById("root");
el2.innerHTML = "hi"; // ERROR: Object is possibly 'null'

12. Enums

Numeric enum (default)

enum Direction {
  Up,    // 0
  Down,  // 1
  Left,  // 2
  Right  // 3
}
const move = Direction.Up;
console.log(move);           // 0
console.log(Direction[0]);   // "Up" — reverse mapping (unique to numeric enums)

String enum

Preferred in real-world code — values are readable strings, not numbers. The course uses string enums (e.g., for thumbnail sizes, notification types).

enum Status {
  Pending  = "PENDING",
  Active   = "ACTIVE",
  Inactive = "INACTIVE",
}
const s: Status = Status.Active;
console.log(s); // "ACTIVE"
// No reverse mapping for string enums

Const enum (inlined at compile time)

const enum Color {
  Red   = "red",
  Blue  = "blue",
}
const c = Color.Red; // compiled to: const c = "red";
// No JavaScript object is generated — purely a compile-time construct
Enum type Values Reverse mapping? JS object generated?

13. Tuples (Fixed Length)

A tuple is an array with a fixed number of elements where each position has a known type.

Example

// Declare a tuple
const point: [number, number] = [10, 20];
const entry: [string, number] = ["age", 30];

// Destructure
const [x, y] = point; // x: number, y: number

// Wrong order is an error
const bad: [string, number] = [30, "age"]; // ERROR

// Optional tuple elements (must be at end)
const opt: [string, number?] = ["hello"]; // second element optional

// Named tuples (TS 4.0+) — documentation only, no runtime effect
type Range = [start: number, end: number];
const r: Range = [0, 100];

// useState in React returns a tuple:
// const [count, setCount] = useState(0); // [number, Dispatch<...>]

14. Abstract Classes & Access Modifiers

Access modifiers

Modifier Accessible in
public (default)Anywhere
privateThe declaring class only
protectedThe declaring class and subclasses
readonlyRead from anywhere, write only in constructor

Example

class Animal {
  public name: string;          // everyone can access
  private secret: string = "!"; // class only
  protected sound: string;       // class + subclasses

  constructor(name: string, sound: string) {
    this.name = name;
    this.sound = sound;
  }

  // Shorthand — declare + assign in constructor params:
  // constructor(public name: string, private id: number) {}
}

class Dog extends Animal {
  constructor(name: string) {
    super(name, "woof");
  }
  speak() {
    return `${this.name} says ${this.sound}`; // protected OK in subclass
    // this.secret // ERROR: private
  }
}

Abstract classes

Cannot be instantiated directly. Subclasses must implement abstract methods.

abstract class Shape {
  abstract getArea(): number; // no implementation — must be provided by subclass

  describe(): string {         // concrete method shared by all subclasses
    return `Area: ${this.getArea()}`;
  }
}

class Circle extends Shape {
  constructor(private radius: number) { super(); }
  getArea(): number { return Math.PI * this.radius ** 2; }
}

// new Shape(); // ERROR: Cannot create an instance of an abstract class
const c = new Circle(5);
console.log(c.describe());  // "Area: 78.53981633974483"
console.log(c.getArea());   // 78.53981633974483

15. Module System & Declaration Files

ES module exports / imports in TypeScript

// math.ts — named exports
export function add(a: number, b: number): number { return a + b; }
export const PI = 3.14159;

// types.ts — exporting types
export interface User { id: number; name: string; }
export type ID = string | number;

// main.ts — imports
import { add, PI } from "./math";
import type { User } from "./types"; // type-only import (erased at compile time)
import { User as UserModel } from "./types"; // alias

Declaration files (.d.ts)

Contain only type information — no executable code. Used to add types for JS libraries that were not written in TypeScript.

// example.d.ts
declare module "my-js-lib" {
  export function doThing(x: string): number;
  export const version: string;
}

// In tsconfig, @types/* packages from DefinitelyTyped are loaded automatically:
// npm install --save-dev @types/node
// npm install --save-dev @types/react

16. tsconfig.json Essentials

Example

{
  "compilerOptions": {
    "target": "ES2017",        // output JS version (default: ES3)
    "module": "commonjs",      // module system (commonjs, ESNext, etc.)
    "strict": true,            // enables ALL strict checks at once
    "noImplicitAny": true,     // error if type is implicitly any
    "strictNullChecks": true,  // null/undefined are not assignable to other types
    "outDir": "./dist",        // compiled JS output directory
    "rootDir": "./src",        // source TypeScript directory
    "esModuleInterop": true,   // enables default imports from CommonJS modules
    "skipLibCheck": true       // skip type-checking declaration files
  },
  "include": ["src/**/*"],
  "exclude": ["node_modules"]
}
Option Effect when true
strictUmbrella flag — enables noImplicitAny, strictNullChecks, strictFunctionTypes, noImplicitThis, and more
noImplicitAnyError when TypeScript infers any for a parameter or variable that has no annotation
strictNullChecksnull and undefined are not assignable to other types — you must handle them explicitly
targetWhich JS version to emit (ES3, ES5, ES2017, ESNext…). Older targets cause downleveling (e.g., template literals → concat())
moduleWhat module syntax to emit (commonjs, ESNext, etc.)

Downleveling: By default TypeScript targets ES3. During compilation, modern syntax (template literals, async/await, optional chaining) is transpiled to older equivalent code. Type annotations are also stripped since browsers cannot execute TypeScript.

Key fact (from slides): Even if TypeScript reports type errors, it still emits the .js file — "TypeScript is built with migration in mind."

17. Common Gotchas

any vs unknown

// any — disables type checking entirely (avoid!)
function badProcess(input: any) {
  input.toUpperCase(); // TypeScript allows this — no safety at all
}

// unknown — type-safe; must narrow before use
function safeProcess(input: unknown) {
  // input.toUpperCase(); // ERROR — cannot use unknown without narrowing
  if (typeof input === "string") {
    console.log(input.toUpperCase()); // OK — narrowed
  }
}
safeProcess("hello"); // "HELLO"
safeProcess(42);      // (no output — not a string)

Type space vs value space

Types and interfaces only exist at compile time (type space). They are erased and cannot be used as runtime values. class and enum exist in both spaces.

interface User { id: number; }

// WRONG — interface is not a value
const x = User; // ERROR: 'User' only refers to a type

// RIGHT — class exists at runtime
class UserClass { id = 0; }
const y = UserClass; // OK — class is a value

// typeof works on values, not types:
type T = typeof UserClass; // the constructor type

Structural typing / duck typing

TypeScript uses structural typing (not nominal). Two types are compatible if they have the same shape — it doesn't matter what they're called or where they were declared. "If it walks like a duck and quacks like a duck, it is a duck."

interface Point { x: number; y: number; }

function logPoint(p: Point) {
  console.log(p.x, p.y);
}

const point3d = { x: 1, y: 2, z: 3 }; // extra property z — still compatible
logPoint(point3d); // OK — has x and y

const animal = { type: "cow", name: "Fred" }; // no x or y
logPoint(animal); // ERROR — missing required properties

// GOTCHA: Excess property check on object literals (stricter than structural typing)
logPoint({ x: 1, y: 2, z: 3 }); // ERROR in object literal — excess property check
// But assigning to a variable first bypasses this (see point3d above)

18. Integration with React

FC and PropsWithChildren

import React, { FC, PropsWithChildren } from "react";

// FC<Props> = FunctionComponent<Props>
interface ButtonProps {
  label: string;
  onClick: () => void;
}
const Button: FC<ButtonProps> = ({ label, onClick }) => (
  <button onClick={onClick}>{label}</button>
);

// PropsWithChildren adds children: React.ReactNode to your props
interface CardProps {
  title: string;
}
const Card: FC<PropsWithChildren<CardProps>> = ({ title, children }) => (
  <div>
    <h2>{title}</h2>
    {children}
  </div>
);

Event types

// Input change event
function handleChange(e: React.ChangeEvent<HTMLInputElement>) {
  console.log(e.target.value); // string
}

// Form submit event
function handleSubmit(e: React.FormEvent<HTMLFormElement>) {
  e.preventDefault();
}

// Button click event
function handleClick(e: React.MouseEvent<HTMLButtonElement>) {
  console.log(e.currentTarget);
}

// Keyboard event
function handleKey(e: React.KeyboardEvent<HTMLInputElement>) {
  console.log(e.key);
}

useState<T>

import { useState } from "react";

const [count, setCount] = useState<number>(0);
const [name, setName] = useState<string>("");
const [user, setUser] = useState<User | null>(null); // nullable state

// TypeScript infers T when initial value is provided:
const [flag, setFlag] = useState(false); // boolean inferred

useRef<T>

import { useRef } from "react";

// DOM element reference — must start as null
const inputRef = useRef<HTMLInputElement>(null);
// inputRef.current is HTMLInputElement | null

// Mutable value reference (not null)
const timerRef = useRef<number | null>(null);

// Usage in JSX:
// <input ref={inputRef} />
// then: inputRef.current?.focus();

Typing component props inline

// Inline prop type (no separate interface needed for simple components)
function Greeting({ name, age }: { name: string; age: number }) {
  return <p>Hello {name}, age {age}</p>;
}

// With children (React 18+)
function Wrapper({ children }: { children: React.ReactNode }) {
  return <div className="wrapper">{children}</div>;
}
