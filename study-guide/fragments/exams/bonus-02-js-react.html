Question 1 — Multiple Choice

What does the following code print to the console?

const obj = {
  name: 'Alice',
  greet: () => {
    console.log(this.name);
  }
};
obj.greet();
  • A) 'Alice'
  • B) undefined
  • C) A ReferenceError
  • D) null
▸ Show Answer

B — undefined

Arrow functions do not have their own this. They capture this from the surrounding lexical scope at the time they are defined. Here the arrow function is defined at the top level (or inside a module), so this is the global object (or undefined in strict mode). Either way, this.name is not 'Alice'. This is the classic "method shorthand vs. arrow function" trap — arrow functions should not be used as object methods when you need this to refer to the object.

Question 2 — What Does This Print?

Predict the exact console output, in order:

function Counter() {

  setInterval(function() {
    this.count++;
    console.log(this.count);
  }, 100);
}

const c = new Counter();
// (after 2 ticks)
▸ Show Answer

Output: NaN, NaN

The callback passed to setInterval is a regular function. When invoked by the timer, this is the global object (or undefined in strict mode), not the Counter instance. this.count on the global object is undefined, so undefined++ yields NaN.

Fix: Use an arrow function instead — setInterval(() => { this.count++; ... }, 100) — so that this is inherited from the constructor's lexical scope (the Counter instance).

Question 5 — Code Analysis

What is the output of the following code? Explain each line.

function Animal(name) {
  this.name = name;
}
Animal.prototype.speak = function() {
  return this.name + ' makes a noise.';
};

function Dog(name) {
  Animal.call(this, name);
}
Dog.prototype = Object.create(Animal.prototype);
Dog.prototype.constructor = Dog;

Dog.prototype.speak = function() {
  return this.name + ' barks.';
};

const d = new Dog('Rex');
console.log(d.speak());
console.log(d instanceof Dog);
console.log(d instanceof Animal);
console.log(Object.getPrototypeOf(d) === Dog.prototype);
▸ Show Answer

Output:

  1. 'Rex barks.' — Dog.prototype.speak overrides Animal.prototype.speak. Method lookup finds the override first.
  2. true — d was created with new Dog, so Dog.prototype is in its chain.
  3. true — Dog.prototype was set to Object.create(Animal.prototype), so Animal.prototype is also in the chain.
  4. true — Object.getPrototypeOf(d) returns Dog.prototype directly.

Key point: Dog.prototype.constructor = Dog is needed because Object.create(Animal.prototype) produces an object whose constructor property points to Animal. Without the reset, d.constructor === Animal would be true instead of Dog.

Question 6 — Multiple Choice

Which of the following correctly describes the three states of a JavaScript Promise?

  • A) loading, success, error
  • B) pending, fulfilled, rejected
  • C) waiting, resolved, failed
  • D) idle, complete, broken
▸ Show Answer

B — pending, fulfilled, rejected

A Promise starts as pending. It transitions to fulfilled when the async operation succeeds (the resolve callback is called) and to rejected when it fails (the reject callback is called). Once settled (fulfilled or rejected), a Promise cannot change state again. The collective term for fulfilled or rejected is settled.

Output (in order):

  1. 1
  2. caught: oops
  3. final: 99

Explanation:

  • The first .then logs 1 and returns 2.

Question 10 — What Does This Print?

Predict the exact console output, in order. This is an event loop ordering puzzle.

console.log('A');

setTimeout(() => console.log('B'), 0);

Promise.resolve().then(() => console.log('C'));

console.log('D');
▸ Show Answer

Output: A, D, C, B

Explanation:

  1. 'A' — synchronous, runs immediately.
  2. setTimeout callback is placed in the macrotask queue.
  3. The Promise.resolve().then callback is placed in the microtask queue.
  4. 'D' — synchronous, runs immediately.
  5. After the synchronous call stack is empty, the event loop drains the microtask queue first → 'C'.
  6. Then it picks the next macrotask → 'B'.

Rule: Microtasks (Promise callbacks, queueMicrotask) always run before the next macrotask (setTimeout, setInterval, I/O callbacks).

Question 11 — What Does This Print?

A more complex event loop puzzle. Predict the output in order:

console.log('start');

setTimeout(() => {
  console.log('timeout 1');
  Promise.resolve().then(() => console.log('micro inside timeout'));
}, 0);

Promise.resolve()
  .then(() => {
    console.log('micro 1');
    return Promise.resolve();
  })
  .then(() => console.log('micro 2'));

setTimeout(() => console.log('timeout 2'), 0);

console.log('end');
▸ Show Answer

Output: start, end, micro 1, micro 2, timeout 1, micro inside timeout, timeout 2

Step-by-step:

  1. 'start' — sync
  2. 'end' — sync
  3. Microtask queue drains: 'micro 1'. Returning Promise.resolve() schedules another microtask.
  4. Microtask queue: 'micro 2'.

Key insight: After each macrotask, the entire microtask queue is drained before moving to the next macrotask.

Question 12 — Multiple Choice

Consider the following async/await code:

async function fetchUser(id) {
  const res = await fetch(`/api/users/${id}`);
  const data = await res.json();
  return data;
}

fetchUser(1).then(u => console.log(u.name));

The server responds with HTTP 404. Which of the following best describes what happens?

  • A) The fetch call throws a network error and fetchUser rejects
  • B) fetch resolves successfully; res.json() might succeed and data could be an error body; no automatic rejection occurs
  • C) res.json() throws because 404 responses have no body
  • D) async functions automatically throw when the response status is not 2xx
▸ Show Answer

B

The fetch API only rejects its Promise on a network failure (no connection, DNS error, etc.). A 404 is a valid HTTP response — fetch resolves with a Response object where res.ok === false and res.status === 404. The server may still send a JSON body (e.g., an error message). You must manually check res.ok and throw if needed:

if (!res.ok) throw new Error(`HTTP error: ${res.status}`);

Question 13 — Code Analysis (Written Short-Answer)

A developer writes the following to make a POST request with a JSON body. Identify ALL mistakes and provide the corrected version.

async function createPost(title, body) {
  const res = await fetch('/api/posts', {
    method: 'POST',
    body: { title, body }
  });
  return res.json();
}
▸ Show Answer

Two mistakes:

  1. Body is not serialized: body must be a string (or a Blob/FormData). Passing a plain object sends [object Object]. Fix: body: JSON.stringify({ title, body }).
  2. Missing Content-Type header: The server needs to know the body is JSON. Fix: add headers: { 'Content-Type': 'application/json' }.

Corrected code:

async function createPost(title, body) {
  const res = await fetch('/api/posts', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ title, body })
  });
  if (!res.ok) throw new Error(`HTTP ${res.status}`);
  return res.json();
}

Question 15 — Written Short-Answer

Explain the difference between interface and type in TypeScript. Give one thing each can do that the other cannot.

▸ Show Answer

Similarities: Both can describe the shape of objects, be used as type annotations, and be extended/intersected.

What interface can do that type cannot:

  • Declaration merging — declaring the same interface name twice causes TypeScript to merge them. This is how library authors augment existing types.
interface User { name: string; }
interface User { age: number; }
// merged: { name: string; age: number }

What type can do that interface cannot:

  • Union and intersection types, mapped types, conditional types, tuples as top-level aliases.
type StringOrNumber = string | number;
type Nullable<T> = T | null;

Common advice: Use interface for object shapes (especially in public APIs), use type for unions, primitives, and complex type expressions.

Question 16 — Multiple Choice

What is the difference between any and unknown in TypeScript?

  • A) They are identical — both opt out of type checking entirely
  • B) any allows all operations without type checking; unknown requires a type check (narrowing) before you can perform operations on the value
  • C) unknown is only for function return types; any is for parameters
  • D) any causes a compile error; unknown is the safe alternative
▸ Show Answer

B

Question 21 — Written Short-Answer

Explain the difference between never and void in TypeScript. When would you use each?

▸ Show Answer

void means a function returns no meaningful value (it returns undefined). It is used for callbacks and functions whose return value is irrelevant.

never means a function never returns at all — it either throws unconditionally, or runs an infinite loop. It is also the type inferred for unreachable code and for the bottom of exhaustive type checks.

// void — returns, but the return value is unused
function log(msg: string): void {
  console.log(msg);
}

// never — throws unconditionally, never returns
function fail(msg: string): never {
  throw new Error(msg);
}

// never — exhaustiveness check
function assertNever(x: never): never {
  throw new Error('Unhandled case: ' + x);
}

Key distinction: A void function reaches its end; a never function cannot.

Question 23 — Code Analysis

Identify the Rules of Hooks violation(s) in the component below and explain why each is a problem:

function UserList({ isAdmin }) {
  const [users, setUsers] = useState([]);

  if (isAdmin) {
    const [adminData, setAdminData] = useState(null);
  }

  for (let i = 0; i < 3; i++) {
    useEffect(() => {
      console.log('effect', i);
    }, []);
  }

  return <div>{users.length}</div>;
}
▸ Show Answer

Two violations:

  1. Hook inside a conditional (if (isAdmin)): useState is called only when isAdmin is true. React relies on the order of hook calls being identical on every render. If isAdmin changes between renders, the hook call order changes, causing React to mismap hook state to the wrong slot — leading to corrupted state or crashes.
  2. Hook inside a loop (for loop): Same principle — useEffect is called a variable number of times. Hook calls must be unconditional and at the top level of the component function.

Fix: Move all hooks to the top level. Use conditional logic inside the hook's callback, not around the hook call itself.

Question 24 — What Does This Print? (React)

What value is alerted when the button is clicked, and why?

function Counter() {
  const [count, setCount] = useState(0);

  const handleClick = () => {
    setTimeout(() => {
      alert(count);
    }, 3000);
    setCount(count + 1);
  };

  return <button onClick={handleClick}>Click me ({count})</button>;
}

The user clicks the button once. Three seconds later, what does the alert show?

▸ Show Answer

The alert shows 0 (the count value at the time the click handler ran).

This is a stale closure. The handleClick arrow function closes over the count variable from the render it was created in. At the time of the click, count is 0. Even though setCount(1) triggers a re-render, the setTimeout callback still holds a reference to the old count = 0 closure, not the new render's count = 1.

Fix: Use a ref to always read the latest value:

const countRef = useRef(count);
useEffect(() => { countRef.current = count; }, [count]);
// then inside setTimeout: alert(countRef.current);

Question 25 — Multiple Choice

A component receives an <input> with both a value prop and an onChange handler. What kind of component is this?

  • A) Uncontrolled — the DOM owns the input's state
  • B) Controlled — React state owns the input's value
  • C) Hybrid — both React and the DOM share ownership
  • D) Ref-controlled — only a ref manages the input
▸ Show Answer

B — Controlled

A controlled component has its form data driven by React state. The value prop locks the input to the state value; the onChange handler updates state in response to user input. This gives React full control over the input's value at all times.

An uncontrolled component uses defaultValue (or no value prop at all) and relies on a ref to read the DOM value imperatively when needed.

Question 26 — Code Analysis

Explain the bug in the following useState usage and provide the correct fix:

function TodoApp() {
  const [todos, setTodos] = useState([]);

  const addTodo = (text) => {
    todos.push({ id: Date.now(), text });
    setTodos(todos);
  };

  return (
    <div>
      {todos.map(t => <p key={t.id}>{t.text}</p>)}
      <button onClick={() => addTodo('new item')}>Add</button>
    </div>
  );
}
▸ Show Answer

Bug: mutating state directly. todos.push(...) mutates the existing array in place. Then setTodos(todos) passes the same array reference. React uses Object.is (reference equality) to decide whether to re-render. Since the reference hasn't changed, React may skip the re-render entirely.

Correct fix — always produce a new array:

const addTodo = (text) => {
  setTodos(prev => [...prev, { id: Date.now(), text }]);
};

Using the functional updater form (prev => ...) is also safer if multiple state updates are batched, as it always operates on the latest state.

Question 28 — Multiple Choice

What is the primary purpose of the key prop in React lists?

  • A) To give each list item a CSS class for styling
  • B) To help React's reconciliation algorithm identify which items have changed, been added, or removed
  • C) To ensure items are rendered in alphabetical order
  • D) To pass a unique identifier to the item's child components
▸ Show Answer

B

The key prop is a hint to the React reconciler. When a list re-renders, React uses keys to match old virtual DOM nodes to new ones. Without stable unique keys, React falls back to index-based matching, which can cause incorrect re-use of component instances when items are reordered, inserted, or deleted — leading to subtle state bugs.

Do not use array index as key when the list can be reordered or items can be inserted/removed. Use a stable, unique ID instead.

Question 29 — Code Analysis

Explain the difference between the two useReducer approaches below and when you'd prefer one over the other:

// Approach A
const [count, setCount] = useState(0);
const increment = () => setCount(count + 1);
const decrement = () => setCount(count - 1);
const reset = () => setCount(0);

// Approach B
const initialState = { count: 0 };
function reducer(state, action) {
  switch (action.type) {
    case 'increment': return { count: state.count + 1 };

Question 32 — Multiple Choice

When should you use React.memo?

  • A) Always — it makes every component faster
  • B) On components that receive the same props frequently and are expensive to re-render, to skip unnecessary re-renders
  • C) Only on class components
  • D) On components that manage their own state and never receive props
▸ Show Answer

B

React.memo wraps a component and performs a shallow prop comparison before re-rendering. If the props haven't changed, the previous render result is reused. It is useful when:

  • The component renders the same output for the same props.
  • The component is re-rendered often by a parent that changes other state.
  • The component is expensive to render (large lists, complex computations).

Not a silver bullet: It adds the overhead of prop comparison on every render. If props change nearly every render, React.memo adds cost without benefit. Always profile first.

      <p>{count}</p>
      <Child onClick={handleClick} />
    </div>
  );
}
▸ Show Answer

Problem: handleClick is a regular function created fresh on every render of Parent. Even though Child is wrapped in React.memo, the shallow prop comparison sees a new function reference for onClick every time — so it re-renders.

Question 34 — Written Short-Answer

▸ Show Answer

Lifting state up means moving shared state to the closest common ancestor of the components that need to read or update it.       <FahrenheitInput
        value={celsius * 9/5 + 32}
        onChange={f => setCelsius((f - 32) * 5/9)}
      />
    </>
  );
}

Question 35 — Code Analysis

What does the following code do? Explain forwardRef and useImperativeHandle.

const FancyInput = React.forwardRef((props, ref) => {
  const inputRef = useRef(null);

  useImperativeHandle(ref, () => ({
    focus: () => inputRef.current.focus(),
    clear: () => { inputRef.current.value = ''; }
  }));

  return <input ref={inputRef} {...props} />;
});

function Form() {
  const fancyRef = useRef(null);
  return (
    <>
      <FancyInput ref={fancyRef} />
      <button onClick={() => fancyRef.current.focus()}>Focus</button>
      <button onClick={() => fancyRef.current.clear()}>Clear</button>
    </>
  );
}
▸ Show Answer

forwardRef allows a parent component to pass a ref down to a child component's DOM element (or to a custom imperative handle). Without it, refs on function components are ignored.

useImperativeHandle customizes what value is exposed to the parent when they hold a ref to the child. Instead of exposing the raw DOM node (which would let the parent do anything), the child exposes only a controlled API — in this case focus() and clear() methods.

Effect: fancyRef.current in Form is the object { focus: fn, clear: fn }, not the raw <input> DOM element. Clicking "Focus" calls the exposed focus method, and clicking "Clear" zeroes the input's value.

When to use: Use useImperativeHandle sparingly — for exposing a minimal imperative API (focus management, animation triggers, scroll commands) while keeping the internal implementation encapsulated.

Question 38 — Written Short-Answer

What is a custom hook in React? Write a simple custom hook called useWindowWidth that tracks the browser window's width and returns it.

▸ Show Answer

A custom hook is a regular JavaScript function whose name starts with use that can call other hooks. Custom hooks let you extract and reuse stateful logic across multiple components without duplicating code or changing the component tree.

function useWindowWidth() {
  const [width, setWidth] = useState(window.innerWidth);

  useEffect(() => {
    const handleResize = () => setWidth(window.innerWidth);
    window.addEventListener('resize', handleResize);
    return () => window.removeEventListener('resize', handleResize);
  }, []); // empty deps — listener registered once, cleaned up on unmount

  return width;
}

// Usage
function MyComponent() {
  const width = useWindowWidth();
  return <p>Window is {width}px wide</p>;
}

Rules: Custom hooks must follow the Rules of Hooks — only call hooks at the top level, never inside conditions or loops.

Question 45 — Written Short-Answer

Describe what happens in each of the four this-binding rules in JavaScript. For each rule, give a one-line code example showing when it applies.

▸ Show Answer

1. Default binding — applies when a function is called as a plain function (no object, no new, no explicit binding). this is the global object (or undefined in strict mode).

function greet() { console.log(this); }
greet(); // window (or undefined in strict)

2. Implicit binding — applies when a function is called as a method on an object. this is the object to the left of the dot.

const obj = { name: 'Bob', greet() { console.log(this.name); } };
obj.greet(); // 'Bob'

3. Explicit binding — applies when .call(), .apply(), or .bind() is used to explicitly set this.

greet.call({ name: 'Carol' }); // 'Carol'

4. new binding — applies when a function is called with the new keyword. A new empty object is created, this is set to it, and the function populates it.

function Person(name) { this.name = name; }
const p = new Person('Dave'); // this === p inside the function

Priority: new > explicit (.call/.bind) > implicit (method) > default.

Question 52 — Written Short-Answer

Explain the difference between Promise.race and Promise.any. Give a practical use case for each.

▸ Show Answer

Promise.race(promises) — settles with whatever promise settles first, whether fulfilled or rejected. If the first promise to settle is rejected, race rejects too.

Use case: Implementing a timeout for a network request:

const result = await Promise.race([
  fetch('/api/data'),
  new Promise((_, reject) =>
    setTimeout(() => reject(new Error('Timeout')), 5000))
]);

Promise.any(promises) — resolves as soon as any one promise fulfills, ignoring rejections. Only rejects (with an AggregateError) if all promises reject.

Use case: Fetching from multiple CDN mirrors and using whichever responds first:

const asset = await Promise.any([
  fetch('https://cdn1.example.com/asset.js'),
  fetch('https://cdn2.example.com/asset.js'),
  fetch('https://cdn3.example.com/asset.js'),
]);

Key distinction: race cares about the first settlement (reject or resolve); any cares only about the first fulfillment.
